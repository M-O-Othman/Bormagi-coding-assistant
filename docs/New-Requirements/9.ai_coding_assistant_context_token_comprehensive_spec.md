# Requirements Specification and Design Recommendations
## Context, Token, and Conversation Management Enhancements for an Existing AI Coding Assistant

**Document Version:** 2.0  
**Status:** Implementation Ready  
**Audience:** Engineering, Architecture, Product, QA, DevOps  
**Primary Goal:** Upgrade an existing AI coding assistant to reduce token consumption, improve context quality, prevent context-window failures, and increase reliability, speed, and cost-efficiency while preserving or improving answer quality.

---

## 1. Executive Summary

This specification defines the functional and non-functional requirements, target architecture, component responsibilities, algorithms, data contracts, rollout plan, telemetry, and acceptance criteria for enhancing an existing AI coding assistant.

The recommendations in this document consolidate recurring best practices observed across mature open-source coding assistants including Aider, Cline, Continue, and Roo Code. The consolidated design intentionally avoids source-specific contradictions by standardizing on the shared patterns that are consistently reinforced across those systems:

1. **Do not send the entire codebase.** Maintain a compact structural model of the repository and retrieve detailed content only when necessary.
2. **Separate stable context from volatile context.** Stable context must be cached, versioned, and reused.
3. **Use mode-specific prompting and budgets.** Planning, editing, reviewing, and debugging have different context requirements and must not use the same prompt shape.
4. **Compact history proactively.** Do not wait for provider hard-limit failures.
5. **Persist decisions and artifacts as structured memory.** Do not rely on raw transcript replay.
6. **Prefer targeted retrieval over transcript accumulation.** Use search, dependency expansion, symbol resolution, and recent activity to select context.
7. **Constrain edit scope and widen read scope.** Full-file editing context must remain narrow; broader awareness should be represented by summaries, maps, and snippets.
8. **Use explicit token accounting and budget enforcement.** Token management is a first-class runtime concern, not an afterthought.
9. **Use layered instruction files instead of repeating guidance in chat.** Project, directory, and path-scoped guidance must be durable, version-controlled, and loaded automatically.
10. **Use deterministic hooks for validation and enforcement.** Formatting, test gates, protected-file checks, and post-compaction reinjection must not depend on the model remembering to do them.
11. **Represent long-horizon work with plan artifacts.** Multi-step tasks must externalize milestones, validation commands, and decision notes rather than relying only on transient reasoning.
12. **Resume work from checkpoints, not only from raw transcript replay.** Plans, approvals, artifacts, and structured memory must survive compaction and session restarts.

This document is intended to be sufficiently detailed to hand directly to implementation teams.

## 2. Problem Statement

The existing coding assistant exhibits one or more of the following issues:

- Excessive token usage on initial or early requests
- Frequent rate-limit or context-window failures
- Large repeated prompts caused by replaying static instructions and long history
- Poor context relevance because too much unrelated code is included
- Weak session continuity across long interactions
- High cost relative to delivered value
- Low reliability in large repositories or monorepos
- Fragile behavior when tool outputs, diffs, and command logs accumulate

The enhancement program defined here addresses these issues without requiring a full rewrite of the assistant.

## 4. Design Principles

The implementation team shall follow these design principles:

1. **Right context beats more context.**
2. **Static information must not be resent if it can be cached or referenced.**
3. **Long-lived knowledge must be extracted into structured state.**
4. **Context selection must be explainable and observable.**
5. **The assistant must degrade gracefully under budget pressure.**
6. **Read-heavy exploration and write-heavy execution must be separated.**
7. **Context management policies must be deterministic enough for testing.**
8. **The system must remain provider-agnostic where practical.**

---

## 5. Key Terms

- **Stable Context:** Context that changes infrequently during a session, such as system rules, coding standards, tool schemas, architecture summary, and repository map slices.
- **Volatile Context:** Current user request, selected code snippets, terminal output, errors, diffs, and recent assistant actions.
- **Repo Map / Codebase Map:** A compact structural representation of the codebase containing file paths, symbols, signatures, and high-value summaries.
- **Structured Memory:** Durable, machine-readable session state such as decisions, plans, unresolved questions, recent edits, and known constraints.
- **Context Compaction:** Replacing long raw history with a targeted summary that preserves essential decisions and unresolved issues.
- **Mode:** Operational state such as Plan, Edit, Debug, Review, or Explain, each with its own prompt template and budget policy.
- **Editable Scope:** Files or ranges the model may change.
- **Reference Scope:** Files, symbols, or summaries visible to the model but not directly editable.

## 6. Consolidated Findings from Reference Systems

The following findings are normalized across the surveyed repositories and their documentation:

### 6.1 Repository-wide awareness should be summary-based
Mature assistants avoid injecting full codebases. Instead, they maintain a concise structural representation of the repository and then retrieve details only for the files most relevant to the task.

### 6.2 Prompt caching should target the stable prefix
Stable components such as system prompts, read-only project guidance, repo maps, and unchanged context should be grouped into a reusable cached prefix.

### 6.3 Context selection must be retrieval-driven
Relevant context should be selected using a mixture of lexical search, semantic search, import/dependency expansion, language-server symbol resolution, recent edits, and current cursor context.

### 6.4 Planning and execution must be separated
A read-only plan mode significantly reduces unnecessary token use and lowers the risk of premature edits. Write-capable edit mode should only include the exact target context needed for code generation.

### 6.5 Durable guidance should live in layered instruction files
Mature assistants increasingly rely on repository-resident instruction files rather than repeatedly re-explaining conventions in each chat turn. These files can exist at global, repository, directory, or path-specific scope and should be concise, durable, and version-controlled.

### 6.6 Capabilities should be loaded progressively
Where assistants support task-specific skills, commands, or agent profiles, they typically avoid loading the full body of every capability into the initial prompt. Lightweight metadata is exposed first; detailed instructions are loaded only when a capability is selected.

### 6.7 Deterministic hooks improve reliability
Formatting, policy enforcement, validation, notifications, and protected-file checks are more reliable when executed by deterministic hooks tied to lifecycle events rather than relying purely on model discretion.

### 6.8 Long-horizon work benefits from explicit plan artifacts
Complex tasks should be represented by explicit planning documents or structured plan objects containing milestones, validations, decision notes, and stop conditions. This reduces context drift and provides resumable state.

### 6.9 Resume and checkpoint behavior should preserve machine-usable state
Robust assistants preserve plan history, approvals, artifacts, and compact structured memory across resumed sessions so that the model does not need the full historical transcript to continue effectively.

### 6.10 Security boundaries must be visible to the context system
Approval policies, protected paths, tool allowlists, and sandbox modes must be represented explicitly in runtime policy so the assistant can plan within constraints instead of discovering them only at execution time.

### 6.5 Long sessions require active compaction
Long-running sessions must summarize older interactions before reaching hard model limits, preserving decisions, plans, unresolved blockers, and important artifacts.

### 6.6 Raw chat history is a weak memory layer
High-performing assistants increasingly preserve durable state in structured forms such as rules, mode state, recent actions, codebase indices, and compact summaries rather than depending entirely on long transcript replay.

### 6.7 Tool outputs must remain machine-usable
Search hits, diagnostics, diffs, test failures, and terminal output should be retained as structured artifacts, not only as natural-language paraphrases.

### 6.8 Edit scope should be constrained
Too many editable files decrease model accuracy and inflate prompt cost. Systems should constrain the editable set and use summaries or narrow snippets for the wider dependency surface.

### 6.9 Context thresholds should be configurable
Different providers and models expose different context windows and rate limits; thresholds for compaction and safety margins must therefore be configurable at provider, model, and mode levels.

## 7. Target Operating Model

The enhanced assistant shall run a **context pipeline** for every request:

1. classify request intent and operating mode
2. load relevant structured memory
3. compute token budget and available margin
4. gather candidate context from codebase index and tools
5. rank and prune candidate context
6. assemble prompt according to mode template
7. apply cache segmentation for stable prefix
8. compact prior history if thresholds are exceeded
9. send model request
10. persist machine-readable outputs, telemetry, and updated memory

This pipeline is mandatory and replaces any previous design that directly concatenates all session content into a single growing transcript.

## 8. Functional Requirements

## FR-1 Request Classification and Mode Selection

### Requirement
The system shall classify each incoming request into one primary operating mode and zero or more secondary intents.

### Supported Primary Modes
- `plan`
- `edit`
- `debug`
- `review`
- `explain`
- `search`
- `test-fix`

### Design Recommendation
Use a lightweight classifier or rules layer before prompt construction. This classifier must not use the premium coding model by default.

### Acceptance Criteria
- At least 95% of benchmark requests are assigned the expected primary mode.
- Mode selection result is visible in telemetry.

### TypeScript Interface

```ts
export type AssistantMode =

  | "search"
  | "test-fix";

export interface ModeDecision {
  mode: AssistantMode;
  confidence: number;
  secondaryIntents: string[];
  reason: string;
}
```

## FR-2 Repository Map / Codebase Structural Index

### Requirement
The system shall maintain a compact, queryable structural representation of the repository.

### Minimum Content
For each indexed file, the system shall store where applicable:
- path
- language
- exported symbols
- classes / methods / functions
- signatures
- top-level comments / docstrings summary
- import/export edges
- file size and line count
- last modified timestamp
- generated-file flag
- test-file flag
- framework tags if inferable

### Design Recommendation
The repo map must be stored separately from raw file text and optimized for low-token serialization. It must support partial serialization by package, directory, or relevance slice.

### Acceptance Criteria
- The system can generate a compact repo map slice for a large repository without loading all full file contents into the prompt.
- Serialization output remains below configurable token budgets.

### TypeScript Interface
```ts
export interface SymbolEntry {
  name: string;
  kind: "class" | "function" | "method" | "interface" | "type" | "const" | "enum";
  signature?: string;
  summary?: string;
  lineStart?: number;
  lineEnd?: number;
}

export interface FileMapEntry {
  path: string;
  language: string;
  exports: string[];
  imports: string[];
  symbols: SymbolEntry[];
  summary?: string;
  lineCount: number;
  byteSize: number;
  lastModifiedUtc?: string;
  flags: {
    generated: boolean;
    test: boolean;
    config: boolean;
    vendored: boolean;
    binary: boolean;
  };
}

export interface RepoMap {
  repoRoot: string;
  generatedAtUtc: string;
  entries: FileMapEntry[];
}
```

### Example Serialization
```ts
export function serializeRepoMapSlice(entries: FileMapEntry[]): string {
  return entries.map(e => {
    const symbols = e.symbols
      .slice(0, 20)
      .map(s => `- ${s.kind} ${s.name}${s.signature ? ` ${s.signature}` : ""}`)
      .join("\n");

    return [
      `FILE: ${e.path}`,
      `LANG: ${e.language}`,
      e.summary ? `SUMMARY: ${e.summary}` : "",
      symbols ? `SYMBOLS:\n${symbols}` : ""
    ].filter(Boolean).join("\n");
  }).join("\n\n");
}
```

## FR-3 File Eligibility and Exclusion Policy

### Requirement
The system shall exclude low-value files from indexing and retrieval by default unless explicitly requested.

### Default Exclusions
- dependency directories
- build outputs
- lockfiles, unless dependency reasoning is requested
- minified assets
- binaries
- generated code
- test snapshots
- huge data dumps
- vendor code

### Design Recommendation
Implement ignore precedence in this order:
1. explicit user include
2. project allowlist
3. project ignore file
4. built-in ignore patterns

### Acceptance Criteria
- Default indexing excludes non-source noise.
- The user or admin can override exclusions for targeted cases.

## FR-4A Durable Instruction Layers and Scoped Guidance

### Requirement
The system shall support durable instruction files that can be loaded automatically from multiple scopes without requiring the user to restate them in chat.

### Mandatory Scopes
- user/global scope
- organization/workspace scope
- repository root scope
- directory subtree scope
- path-specific scope
- mode-specific instruction overlays where applicable

### Design Recommendation
Instruction files shall remain concise, reviewable, and version-controlled. The system should merge instruction layers from broadest to narrowest scope, with the nearest applicable scope taking precedence for files under that subtree. Instructions should be injected as part of the stable prefix, not as repeated volatile transcript text.

### Normative Rules
- The instruction resolver shall walk from workspace root toward the active file path and collect applicable guidance.
- Path-specific instructions shall be preferred over repository-wide guidance for directly matched files.
- Instruction loading shall be bounded by a configurable token cap.
- Each resolved instruction block shall retain source metadata so the assistant can explain why it was applied.
- The system shall expose an `effectiveInstructions` diagnostic view for debugging.

### Acceptance Criteria
- The assistant applies repository-wide instructions without the user repeating them in the prompt.
- When a file matches both repository-wide and path-specific instructions, both are merged in the configured precedence order.
- The resolved instruction set is visible in telemetry and can be reproduced in tests.

### TypeScript Interfaces
```ts
export interface InstructionLayer {
  id: string;
  scope: "global" | "workspace" | "repo" | "directory" | "path" | "mode";
  sourcePath: string;
  appliesTo: string[];
  priority: number;
  content: string;
  tokenEstimate: number;
}

export interface EffectiveInstructions {
  layers: InstructionLayer[];
  mergedText: string;
  tokenEstimate: number;
}

export function resolveInstructionLayers(args: {
  workspaceRoot: string;
  activePaths: string[];
  mode: AssistantMode;
  maxTokens: number;
}): EffectiveInstructions {
  // Implementation shall collect from broad to narrow scope,
  // prune by relevance and token cap, and preserve provenance.
  return { layers: [], mergedText: "", tokenEstimate: 0 };
}
```

## FR-4B Progressive Capability Loading, Skills, and Agent Profiles

### Requirement
The system shall support task-specific capabilities such as skills, commands, or agent profiles without loading the full body of every capability into the initial prompt.

### Design Recommendation
Capabilities should be registered with lightweight metadata that is always available, while heavy instructional content and supporting assets are loaded only when a capability is selected by the planner, invoked explicitly by the user, or required by policy.

### Normative Rules
- Capability discovery metadata shall include `name`, `description`, `applicableModes`, `requiredTools`, and approximate token cost.
- Full capability instructions shall be loaded lazily.
- The planner shall consider capability cost as part of context budgeting.
- Capability activation events shall be recorded in telemetry and session memory.

### Acceptance Criteria
- A session with 20 installed capabilities does not load all 20 full instruction bodies on startup.
- The assistant can activate a capability, pull its instructions, execute the related task, and persist the activation in checkpoint state.

### TypeScript Interfaces
```ts
export interface CapabilityManifest {

  scripts?: string[];
}

export async function maybeLoadCapability(
  manifests: CapabilityManifest[],
  request: RetrievalQuery,
  budget: number

  // Rank capabilities by relevance and load at most the minimum needed.
  return null;
}
```

## FR-5 Context Retrieval Pipeline

### Requirement
The system shall retrieve context using multiple signals and shall rank candidate items before prompt assembly.

### Required Retrieval Signals
- current file / active selection
- lexical search
- semantic search over embeddings or equivalent index
- import / dependency graph expansion
- language-server definitions and references where available
- recent edit recency
- diagnostics / test failures / stack traces
- user-explicit attachments or mentions
- repo map slice

### Design Recommendation
Adopt a retrieval pipeline with deterministic scoring and explainable weights. Retrieval shall support both full-file and snippet-level outputs.

### Acceptance Criteria
- Retrieval telemetry exposes why top context items were selected.
- The assistant can answer code questions with fewer irrelevant files than the current baseline.

### TypeScript Interface
```ts
export interface RetrievalQuery {
  text: string;
  mode: AssistantMode;
  activeFile?: string;
  selectedSymbol?: string;
  stackTrace?: string;
  failingTestNames?: string[];
}

export interface ContextCandidate {
  id: string;
  kind: "file" | "snippet" | "symbol" | "tool-output" | "memory" | "repo-map";
  path?: string;
  symbol?: string;
  content: string;
  tokenEstimate: number;
  score: number;
  reasons: string[];
  editable: boolean;
}
```

### Ranking Example
```ts
export function scoreCandidate(input: {
  lexical: number;
  semantic: number;
  importProximity: number;
  recency: number;
  activeFileBoost: number;
  diagnosticsBoost: number;
  explicitMentionBoost: number;
}): number {
  return (
    input.lexical * 0.22 +
    input.semantic * 0.24 +
    input.importProximity * 0.16 +
    input.recency * 0.10 +
    input.activeFileBoost * 0.10 +
    input.diagnosticsBoost * 0.10 +
    input.explicitMentionBoost * 0.08
  );
}
```

## FR-6 Mode-Specific Prompt Templates

### Requirement
The system shall use different prompt templates, context priorities, and token budgets for different modes.

### Mode Rules

#### Plan Mode
- read-only tools only
- prefer repo map, symbols, search hits, summaries, architecture notes
- avoid loading many full files
- output plan, assumptions, impacted files, risks, and next action recommendation

#### Edit Mode
- include exact target file(s) and narrow dependency context
- include edit ranges when available
- keep editable file count under configurable threshold

#### Debug Mode
- prioritize stack traces, failing test output, diagnostics, recent diffs, and implicated symbols

#### Review Mode
- prioritize git diff, changed files, tests, architectural rules, and style constraints

#### Explain/Search Mode
- prioritize high-level summaries, definitions, references, and narrow code excerpts

### Acceptance Criteria
- Each mode has a tested prompt template.
- Mode-specific token budgets are enforced.

## FR-7 Token Estimation and Preflight Budget Enforcement

### Requirement
The system shall estimate input token usage before every model request and shall enforce per-mode and per-model budget constraints.

### Design Recommendation
Token estimates may be approximate, but the policy engine must reserve safety margin for provider framing overhead, tool schemas, retries, and response generation.

### Enforcement Rules
- If budget is exceeded, the system shall prune low-ranked candidates.
- If still exceeded, the system shall compact history.
- If still exceeded, the system shall reduce repo map granularity.
- If still exceeded, the system shall shift to degraded mode and explain the reduction internally in telemetry.

### Acceptance Criteria
- Provider errors due to prompt oversize are reduced below target threshold after rollout.

### Example
```ts
export interface BudgetCheckResult {
  fits: boolean;
  estimatedInputTokens: number;
  hardLimit: number;
  softLimit: number;
  overflowBy: number;
  actions: string[];
}

export function checkBudget(
  estimatedInputTokens: number,
  hardLimit: number,
  safetyMargin: number
): BudgetCheckResult {
  const softLimit = Math.max(0, hardLimit - safetyMargin);
  const fits = estimatedInputTokens <= softLimit;
  return {
    fits,
    estimatedInputTokens,
    hardLimit,
    softLimit,
    overflowBy: fits ? 0 : estimatedInputTokens - softLimit,
    actions: fits ? [] : ["prune-context", "compact-history", "reduce-repo-map"]
  };
}
```

## FR-8 Stable Prefix Caching

### Requirement
The system shall separate the prompt into a stable cached prefix and a volatile suffix.

### Stable Prefix Candidates
- system prompt
- coding policies
- project-level rules
- tool schemas
- repo summary or repo map slice
- structured memory summary

### Volatile Suffix Candidates
- current request
- selected files/snippets
- command output
- diagnostics
- current diff
- recent conversation tail

### Design Recommendation
Stable prefix cache invalidation must be based on hashes of component content and relevant version metadata.

### Acceptance Criteria
- Repeated calls within a session reuse stable context rather than reserializing it in full.
- Cache hit rate is measurable.

### Example
```ts
export interface CachedPromptSegment {
  cacheKey: string;
  content: string;
  contentHash: string;
  createdAtUtc: string;
  componentType: "system" | "rules" | "repo-map" | "memory" | "tools";
}

export function buildCacheKey(parts: string[]): string {
  return parts.join("::");
}
```

## FR-9 Conversation Tail and Compaction

### Requirement
The system shall compact conversation history before exceeding configured thresholds.

### Compaction Thresholds
Each model profile shall support:
- warning threshold
- prune threshold
- compact threshold
- emergency degradation threshold

### Compaction Summary Must Preserve
- current task objective
- important decisions
- unresolved issues
- recent file changes
- pending next actions
- failed attempts that must not be repeated blindly

### Design Recommendation
Use a dedicated compaction path or smaller model where possible. The summary must be structured first, optionally followed by a human-readable narrative.

### Acceptance Criteria
- Long sessions remain operable without catastrophic context resets.
- The quality of post-compaction continuation remains within agreed regression thresholds.

## FR-11 Tool Output Normalization

### Requirement
The system shall store and reuse tool outputs as structured artifacts.

### Covered Tool Outputs
- grep/search hits
- build output
- linter output
- test failures
- stack traces
- git diff metadata
- command summaries

### Design Recommendation
Normalize tool outputs into bounded-size schemas. Large raw logs shall be truncated, chunked, and summarized while preserving source links and line references.

### Acceptance Criteria
- Follow-up model calls can consume concise structured tool outputs instead of large raw logs.

### Example
```ts
export interface TestFailureArtifact {
  suite: string;
  testName: string;
  message: string;
  file?: string;
  line?: number;
```

Determines primary mode and task characteristics.

### 11.2 Budget Engine

## 12. Retrieval and Ranking Design

The retrieval layer shall implement a **candidate generation + ranking + pruning** flow.

### 12.1 Candidate Sources
- active editor file and selection
- recent files edited in this session
- lexical search hits
- semantic search hits
- import graph neighbors
- symbol definitions and references
- diagnostics-linked files
- changed files in current diff
- repo map slice for impacted directories
- structured memory items

### 12.2 Ranking Heuristics
The ranking function should consider:
- direct mention in user prompt
- symbol match confidence
- failure stack trace proximity
- active file boost
- recent edit boost
- test failure linkage
- architectural relevance tags
- token cost penalty

### 12.3 Snippet Extraction Rules
When a full file is not essential, extract bounded snippets around:
- symbol definitions
- failing lines
- referenced call sites
- diff hunks

### Snippet Example
```ts
export interface SnippetWindow {
  startLine: number;
  endLine: number;

export function boundedWindow(line: number, before = 30, after = 40): SnippetWindow {
  return {
    startLine: Math.max(1, line - before),
    endLine: line + after
  };
}
```

## 17. Error Handling and Recovery

### Required Behaviors
- detect input over-budget before provider call
- parse provider rate-limit feedback when available
- apply exponential backoff where retries are appropriate
- avoid retrying unchanged oversized requests
- switch to reduced-context recovery path where necessary

### Example
```ts
export async function withBackoff<T>(fn: () => Promise<T>, maxAttempts = 4): Promise<T> {
  let attempt = 0;
  let delayMs = 500;
  while (true) {
    try {
      return await fn();
    } catch (err) {
      attempt++;
      if (attempt >= maxAttempts) throw err;
      await new Promise(res => setTimeout(res, delayMs));
      delayMs *= 2;
    }
  }
}
```

### Mandatory Rule
The system shall not perform identical immediate retries for requests already known to exceed token budget.

## 23. Reference Implementation Sketch

```ts
export async function handleAssistantRequest(input: {
  userText: string;
  activeFile?: string;
  selectedText?: string;
  diagnostics?: string[];
}) {
  const modeDecision = await classifyMode(input.userText, input);
  const profile = await getActiveModelProfile(modeDecision.mode);
  const memory = await loadSessionMemory();

  const retrievalQuery: RetrievalQuery = {
    text: input.userText,
    mode: modeDecision.mode,
    activeFile: input.activeFile
  };

  let candidates = await retrieveCandidates(retrievalQuery, memory);
  candidates = rankAndPruneCandidates(candidates, modeDecision.mode, profile);

  let envelope = await buildContextEnvelope(candidates, modeDecision.mode);
  let prompt = await assemblePrompt({
    mode: modeDecision.mode,
    memory,
    envelope,
    userText: input.userText,
    profile
  });

  let estimate = estimateTokens(prompt) + profile.estimatedToolOverheadTokens;
  let budgetCheck = checkBudget(estimate, profile.maxContextTokens, 3000);

  if (!budgetCheck.fits) {
    const compacted = await compactIfNeeded();
    envelope = await reduceEnvelope(envelope, compacted);
    prompt = await assemblePrompt({
      mode: modeDecision.mode,
      memory,
      envelope,
      userText: input.userText,
      profile
    });
  }

  const response = await callProvider(prompt, profile);
  await persistArtifactsAndMemory(response, envelope, modeDecision.mode);
  return response;
}
```

## 24. Detailed Design Recommendations by Feature Area

### 24.1 Repo Map Granularity
Do not generate one monolithic repo map string for all use cases. Support:
- whole-repo map for tiny projects
- directory/package map slices for medium repos
- impacted-subtree maps for large repos and monorepos

### 24.2 Embedding Search vs Lexical Search
Use hybrid retrieval. Lexical search performs better for exact symbols, error strings, filenames, and config keys; semantic search helps when the user describes behavior rather than exact identifiers.

### 24.3 Prompt Contracts
Require output contracts by mode. Examples:
- Plan mode: assumptions, impacted files, plan steps, risks
- Edit mode: patch summary, changed files, validation notes
- Debug mode: root cause hypothesis, evidence, proposed fix
- Review mode: findings by severity, suggested changes, confidence

### 24.4 Session Hygiene
A long-lived session should not become a dumping ground for every prior log and explanation. The assistant must continuously curate session state.

### 24.5 Monorepo Handling
Monorepo support should include workspace segmentation, package ownership metadata, and subtree-constrained retrieval.

### 24.6 Cross-Repository Reference
When the assistant needs awareness of related repositories, it should prefer imported summaries or repo maps from those repositories rather than full raw code inclusion.

## 27. Risks and Mitigations

| Risk | Description | Mitigation |
|---|---|---|
| Over-pruning | Important context may be excluded | Use retrieval telemetry, allow user pinning, keep active-file priority |
| Weak compaction | Summary may omit critical facts | Preserve structured decisions, blockers, recent edits, next actions |
| Stale repo map | Structural index becomes outdated | Incremental refresh on file change and periodic background rebuild |
| Ranking drift | Heuristics degrade on unusual repos | Make weights configurable and benchmarked |
| Cache misuse | Stale stable prefix used after changes | Strict hash-based invalidation |
| Provider mismatch | Assumptions differ across vendors | Model profile abstraction and provider adapters |

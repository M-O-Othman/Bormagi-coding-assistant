# 1. Executive Summary

This document defines the architecture for adding a knowledge base layer, long-term memory, multi-agent collaboration, and tool use to an existing AI agent framework that uses SOUL/IDENTITY files to define agent personas.

The system targets self-hosted, local-first deployment.ChromaDBprovides vector storage. The implementation language is TypeScript running on Node.js. The design supports multiple LLM providers but defaults to Claude API.

Priority order:(1) Multi-agent collaboration,
(2) Long-term agent memory, (3) File-based RAG, (4) Tool use / function calling.

## 4.5 RAG Pipeline

The retrieval-augmented generation pipeline executes at query time. It retrieves relevant knowledge chunks and injects them into the LLM prompt alongside the agent identity and memory context.

### 4.5.1 Query Flow

1.Agent receives a user message or task.

2.The RAG routeranalyzesthe message to determine which knowledge collections to query.

3.The query is embedded using the same model that indexed the collection.

4.ChromaDBreturns top-K results (default K=5, configurable per collection).

5.Results are re-ranked by a lightweight cross-encoder or LLM-basedreranker.

6.Top results are formatted and injected into the prompt under a <knowledge_context> section.

7.The LLM generates a response grounded in the retrieved context.

### 4.5.2 Prompt Assembly Order

The final prompt sent to the LLM follows this exact assembly order:

SYSTEM PROMPT:

1. SOUL.md content
(personality, constraints)

2. IDENTITY.md content
(role, responsibilities)

3. <agent_memory> (summarized episodic + semantic memory)

4. <knowledge_context> (directly injected critical docs)

5. <retrieved_knowledge> (RAG results, ranked)

6. <available_tools> (tool definitions for function calling)

7. <collaboration_context> (messages from other agents, if any)

USER MESSAGE:

8. The actual user query or task

### 4.5.3 Retrieval Optimization

Three techniques improve retrieval quality beyond naive vector similarity:

Query Decomposition.For complex queries, decompose into sub-queries. Example: 'How does the payment service authenticate requests and what are the rate limits?' becomes two queries: 'payment service authentication' and 'payment service rate limits'. Merge and deduplicate results.

Hypothetical Document Embeddings (HyDE).Generate a hypothetical answer to the query using the LLM, then embed that answer and use it as the search vector. This closes the semantic gap between short queries and long document chunks. Adds one LLM call
(~200ms). Use for ambiguous or abstract queries.

Metadata Filtering.Before vector search, apply metadata filters to narrow the search space. Examples: filter bycontent_type='code' for coding questions, filter bydecision_status='accepted' for architecture queries, filter byupdated_at> '2026-01-01' for recent changes.

# 5. Long-Term Agent Memory (L3)

Memory gives agents continuity across conversations. Without it, each interaction starts from zero. The memory system has two components: episodic memory (what happened) and semantic memory (what was learned).

## 5.1 Episodic Memory

Episodic memory stores summaries of past conversations and interactions. After each conversation, the system generates a structured summary.

interfaceEpisodicMemory{

id: string;

agent_id: string;

session_id: string;

timestamp: string;

summary: string;// LLM-generated conversation summary

key_decisions: string[];// decisions made during this session

action_items:ActionItem[];// tasks created or updated

entities_mentioned: string[];// people, systems, concepts referenced

sentiment: string;// 'positive' | 'neutral' |
'negative' | 'escalated'

token_count: number;// summary size

}

interfaceActionItem{

description: string;

assignee?: string;

status: 'open' | 'in_progress' | 'done';

due_date?: string;

}

Summaries are stored as YAML files in the agent's memory/episodic/ directory and indexed inChromaDBfor retrieval. The system keeps the last
10 summaries in direct context and retrieves older ones via RAG when relevant.

## 5.2 Semantic Memory

Semantic memory stores distilled facts and relationships extracted from conversations and documents. Think of it as the agent's accumulated understanding.

interfaceSemanticFact{

id: string;

agent_id: string;

fact: string;// 'User prefers Terraform overPulumi'

confidence: number;// 0.0 to 1.0

source_type: string;// 'conversation' |
'document' | 'decision' | 'observation'

source_id: string;// reference to origin

created_at: string;

last_validated: string;// when this fact was last confirmed

contradicts?: string[];// IDs of facts this supersedes

tags: string[];

}

The memory manager runs a consolidation process after each session: extract new facts, check for contradictions with existing facts, update confidence scores, and prune low-confidence facts older than 30 days.

## 5.3 Decision Log

Decisions are first-class objects. Every significant decision is recorded with full context so agents (and humans) can understand why something was decided.

interfaceDecisionRecord{

chosen_option: string;

rationale: string;// why this option was chosen

}

}

## 6.3 Task Delegation Protocol

When an agent encounters work outside its capabilities, it delegates to the most appropriate agent using this protocol:

1.The delegating agent checks the agent registry for agents with matching capabilities.

2.It constructs atask_delegationmessage with: task description, required context (knowledge chunks), acceptance criteria, deadline.

3.The receiving agent can accept, reject (with reason), or counter-propose (suggest modifications).

4.On acceptance, the receiving agent works the task and returns atask_resultmessage.

5.The delegating agent validates the result against acceptance criteria.

6.If validation fails, the task is returned with feedback. Max 3 retry cycles before human escalation.

## 6.4 Shared Knowledge Protocol

Agents can share knowledge in two ways:

Push model.When an agent makes a discovery or decision relevant to others, it sends aknowledge_sharemessage. Example: the architect decides on a new API contract; it pushes this to backend-dev and frontend-dev.

Pull model.An agent sends aknowledge_requestto another agent (or broadcast). Example: backend-dev needs to know the database schema constraints; it queries the architect.

Shared knowledge is written to the shared/knowledge/ directory and indexed in sharedChromaDBcollections that all agents can query.

## 6.5 Conflict Resolution

When agents disagree (e.g., the architect proposes a schema that the backend-dev considers impractical), the system follows an escalation ladder:

1.Direct negotiation: agents exchange arguments for 2 rounds max.

2.Mediation: a designated mediator agent (or the architect for technical disputes) reviews both positions.

3.Human escalation: if no agreement after mediation, the conflict is flagged for human review with a structured summary of both positions.

## 7.2 Tool Execution Runtime

Tools run in a sandboxed TypeScript runtime. The executor handles parameter validation, timeout enforcement, error capture, and result formatting.

interfaceToolExecutionResult{

tool_name: string;

success:boolean;

result?: unknown;

error?: { code: string; message: string; recoverable:boolean};

duration_ms: number;

tokens_used?: number;// if tool involved an LLM call

}

Safety constraints: all file operations are restricted to the project directory (no escape). Database queries are read-only unless explicitly granted write access in the manifest. External API calls go through the MCP server which enforces rate limits and authentication.

## 8.6 Document Chunking Service

//src/knowledge/chunker.ts

import {encoding_for_model} from 'tiktoken';

interfaceChunkOptions{

chunkSize: number;// in tokens

chunkOverlap: number;// in tokens

contentType: string;

}

export classDocumentChunker{

private encoder =encoding_for_model('cl100k_base');

chunk(content: string, options:ChunkOptions): { text: string; index: number
}[] {

switch (options.contentType) {

case 'markdown': returnthis.chunkMarkdown(content, options);

case 'code': returnthis.chunkCode(content, options);

default: returnthis.chunkByTokens(content, options);

}

}

privatechunkMarkdown(content: string, opts:ChunkOptions)
{

// Split on headings first

constsections =content.split(/^(#{1,6}\s.+)$/m);

constchunks: { text: string; index: number }[] = [];

letcurrentChunk= '';

letidx= 0;

for (constsection of sections) {

constcombined =currentChunk+ '\n' + section;

if (this.tokenCount(combined) >opts.chunkSize&&currentChunk) {

chunks.push({ text:currentChunk.trim(), index:idx++ });

// Keep overlap from end of previous chunk

constoverlapText=this.getLastNTokens(currentChunk,opts.chunkOverlap);

currentChunk=overlapText+ '\n'
+ section;

} else {

currentChunk= combined;

}

}

if (currentChunk.trim()) {

chunks.push({ text:currentChunk.trim(), index:idx});

}

return chunks;

}

privatechunkCode(content: string, opts:ChunkOptions)
{

// Split on function/class boundaries

// For production: use tree-sitter for AST-aware splitting

constfunctionPattern=
/^(export\s+)?(async\s+)?function\s+|^(export\s+)?(const|let)\s+\w+\s*=\s*(async\s+)?\(/m;

constclassPattern=
/^(export\s+)?(abstract\s+)?class\s+/m;

constlines =content.split('\n');

constchunks: { text: string; index: number }[] = [];

letcurrentChunk= '';

letidx= 0;

for (constline of lines) {

if ((functionPattern.test(line) ||classPattern.test(line))

&&this.tokenCount(currentChunk)
>opts.chunkSize/ 2) {

if (currentChunk.trim()) {

chunks.push({ text:currentChunk.trim(), index:idx++ });

}

currentChunk= line + '\n';

} else {

currentChunk+= line + '\n';

if (this.tokenCount(currentChunk)
>=opts.chunkSize) {

chunks.push({ text:currentChunk.trim(), index:idx++ });

currentChunk=this.getLastNTokens(currentChunk,opts.chunkOverlap)
+ '\n';

}

}

}

if (currentChunk.trim())chunks.push({ text:currentChunk.trim(), index:idx});

return chunks;

}

privatechunkByTokens(content: string, opts:ChunkOptions)
{

constsentences =content.split(/(?<=[.!?])\s+/);

constchunks: { text: string; index: number }[] = [];

letcurrentChunk= '';

letidx= 0;

for (constsentence of sentences) {

if (this.tokenCount(currentChunk+ sentence) >opts.chunkSize&&currentChunk) {

chunks.push({ text:currentChunk.trim(), index:idx++ });

currentChunk=this.getLastNTokens(currentChunk,opts.chunkOverlap)
+ ' ' + sentence;

} else {

currentChunk+= ' ' + sentence;

}

}

if (currentChunk.trim())chunks.push({ text:currentChunk.trim(), index:idx});

return chunks;

}

privatetokenCount(text: string): number {

returnthis.encoder.encode(text).length;

}

privategetLastNTokens(text: string, n: number): string {

consttokens =this.encoder.encode(text);

returnthis.encoder.decode(tokens.slice(-n));

}

}

.sort((a, b) => a[1].priority - b[1].priority);

# 12. Security Considerations

All agent knowledge directories must be readable only by the agent runtime process. Use filesystem permissions (chmod700) on agent directories.

ChromaDBshould not be exposed outside localhost in local deployments. In production, put it behind authentication (Chroma supports static token auth).

LLM API keys must be stored in environment variables or a secrets manager, never in config files.

Tool execution is sandboxed: file operations restricted to project directory, database queries read-only by default, external API calls go through the MCP server which enforces allowlists.

Inter-agent messages are trusted within the system boundary. If agents are distributed across hosts, add message signing (HMAC-SHA256) with a shared secret.

Log sanitization: strip PII and credentials from structured logs before persisting.

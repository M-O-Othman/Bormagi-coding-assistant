# AI Coding Assistant — Git and GitHub Integration Requirements Specification

**Document Status:** Ready for implementation  
**Audience:** Engineering, platform, IDE-extension, backend, DevOps, QA, and security teams  
**Scope:** Enhancements to an existing coding assistant to implement robust, safe, and efficient Git and GitHub integration, normalized from best practices observed across OpenAI Codex, Claude Code, GitHub Copilot, Aider, Cline, Continue, and Roo Code.

---

## 1. Purpose

This document defines the functional requirements, non-functional requirements, design recommendations, workflows, data contracts, safety controls, and implementation guidance for adding or strengthening Git and GitHub integration in an existing AI coding assistant.

The target outcome is an assistant that can work effectively inside Git repositories, understand branch and diff context, make reversible code changes, generate high-quality commits and pull request artifacts, run repository-aware validation, and integrate safely with GitHub workflows without compromising developer control, repository integrity, or security.

This specification is intentionally provider-agnostic. Where source systems differ, this document adopts a consistent normalized design rather than mirroring any single product.

---

## 2. Design goals

The enhanced assistant shall:

1. Treat Git as the primary state-management and reversibility mechanism for AI-driven code changes.
2. Treat GitHub as the primary collaboration, review, and automation surface.
3. Preserve human control over destructive or high-impact actions.
4. Keep local experimentation cheap by making rollback easy.
5. Use branch, diff, status, and repository instructions as first-class context inputs.
6. Separate project memory from repository history.
7. Integrate validation into the edit loop so the assistant can verify before proposing completion.
8. Support both local workflows and PR-centric workflows.
9. Avoid mixing unrelated human changes with AI changes.
10. Remain robust under partial connectivity, limited permissions, and enterprise policy constraints.

---

## 3. Normalized cross-source design principles

The following principles are adopted because they recur across the studied systems, even when their implementation details differ.

### 3.1 Use repository-scoped instruction layers
The assistant shall load durable repository instructions from files in the repository, with support for both repository-wide and path-scoped instructions.

### 3.2 Use Git state as context, not just files
The assistant shall understand branch, diff, changed files, staged files, untracked files, merge conflict state, and clean/dirty working tree state.

### 3.3 Make AI changes reversible by default
The assistant shall checkpoint or commit before and/or around AI modifications so that the user can revert changes without losing conversation state.

### 3.4 Keep human and AI changes separable
If the repository contains uncommitted user edits, the assistant shall preserve or isolate them before applying AI edits.

### 3.5 Validate after changes
The assistant shall run linting, tests, formatting, and selected policy checks after edits and feed compact results back into the loop.

### 3.6 Optimize for PR-based collaboration
The assistant shall generate branch-aware summaries, commit messages, PR descriptions, and review-ready outputs.

### 3.7 Respect approval and sandbox boundaries
Git and GitHub actions with side effects shall be gated by explicit policy, approval, and environment capability.

### 3.8 Preserve reasoning artifacts separately from Git history
Plans, memory, decisions, and task state shall not depend solely on commit history.

## 6. Terminology

- **Assistant:** the existing coding assistant being enhanced.
- **Workspace repository:** the Git repository containing the user’s active project.
- **Primary repository:** the repository that owns the current working directory.
- **Checkpoint:** a reversible snapshot created by the assistant before or during changes.
- **Auto-commit:** a commit created automatically by the assistant.
- **Shadow repository:** an internal Git-backed checkpoint store kept separate from the user-visible repository history.
- **Instruction layer:** repository-wide or path-specific durable guidance files.
- **Validation loop:** execution of format/lint/test/policy commands after edits, optionally followed by automated repair.
- **PR artifact:** generated pull request title, body, checklist, linked issues, and summary.

## 8. Core functional requirements

## 8.1 Repository discovery and root detection

### FR-001 Repository detection
The assistant shall detect the active Git repository by walking upward from the current working file or terminal directory until it finds a repository root marker.

### FR-002 Root markers
The implementation shall support configurable root markers, with `.git` as the default and optional support for additional markers.

### FR-003 Multi-repository workspaces
If the workspace contains multiple repositories, the assistant shall bind the current task to the nearest repository unless the user explicitly selects another repository.

### FR-004 Detached and unsupported states
The assistant shall detect detached HEAD state, bare repositories, submodules, and worktrees and expose those states to the orchestration layer.

### FR-005 Repository capability object
The assistant shall materialize a normalized repository capability object before any Git/GitHub action is attempted.

Example:

```ts
export interface RepoCapabilities {
  isGitRepo: boolean;
  rootPath: string;
  headRef: string | null;
  isDetachedHead: boolean;
  isWorktree: boolean;
  hasGitHubRemote: boolean;
  remotes: Array<{ name: string; url: string }>;
  canCommit: boolean;
  canCreateBranch: boolean;
  canPush: boolean;
  canOpenPullRequest: boolean;
  sandboxAllowsGitWrite: boolean;
}
```

## 8.2 Working tree and branch awareness

### FR-006 Working tree status ingestion
The assistant shall read and normalize:
- staged changes
- unstaged changes
- untracked files
- ignored files where needed
- merge conflict state
- ahead/behind counts where available

### FR-007 Branch context as first-class prompt input
The assistant shall expose the current branch name, base branch if inferable, and changed file set to the prompt assembly layer.

### FR-008 Diff context mode
The assistant shall support a diff-only context mode for tasks such as review, summarization, commit message generation, and PR drafting.

### FR-009 Merge conflict detection
If merge conflicts exist, the assistant shall enter conflict-resolution mode and avoid unrelated code changes until conflicts are resolved or explicitly bypassed.

### FR-010 Dirty-tree classification
The assistant shall classify the working tree into one of these states:
- clean
- dirty-user-only
- dirty-assistant-only
- mixed
- conflicted

Example:

```ts
export type WorkingTreeState =
  | "clean"
  | "dirty-user-only"
  | "dirty-assistant-only"
  | "mixed"
  | "conflicted";
```

---

## 8.3 Safe handling of dirty repositories

### FR-011 Preserve pre-existing user changes
If the assistant is about to edit files in a dirty repository, it shall preserve pre-existing user changes before introducing AI changes.

### FR-012 Isolation strategies
The implementation shall support at least one of the following strategies and should support both:
1. pre-AI protective commit in the main repo
2. checkpoint in a shadow repository with restoration metadata

### FR-013 Explicit policy for dirty-tree behavior
Administrators or users shall be able to configure behavior for dirty trees:
- refuse edits
- auto-checkpoint
- auto-commit protective snapshot
- ask for approval

Recommended normalized default:
- create a reversible checkpoint automatically
- require approval before creating a visible Git commit in the main repository

### FR-014 Protect unrelated files
When possible, the assistant shall avoid touching files already modified by the user unless the task explicitly targets them or requires them.

---

## 8.4 Checkpoints and reversibility

### FR-015 Automatic checkpoint creation
The assistant shall create a checkpoint at task start and before applying a batch of file modifications.

### FR-016 Checkpoint restore
The assistant shall support restoring any checkpoint while preserving conversation and task memory.

### FR-017 Checkpoint visibility
The UI shall clearly display available checkpoints and their reason for creation.

### FR-018 Checkpoint implementation modes
The system shall support:
- **visible Git mode:** commits in the working repository
- **shadow Git mode:** commits in a hidden or auxiliary repository

### FR-019 Binary and rename support
Checkpoints shall capture file adds, deletions, renames, and binary file changes where technically feasible.

Recommended design:
- default to **shadow Git checkpoints** for safety and low-noise experimentation
- optionally offer **main-repo auto-commits** for terminal-first or advanced users

Example:

```ts
export interface Checkpoint {
  id: string;
  repoRoot: string;
  createdAt: string;
  trigger: "task_start" | "before_edit" | "before_command" | "manual";
  storageMode: "shadow" | "main-repo";
  label: string;
  affectedPaths: string[];
}
```

## 8.5 File edit lifecycle

### FR-020 Pre-edit pipeline
Before file edits, the assistant shall perform:
1. repository state refresh
2. instruction layer resolution
3. checkpoint creation
4. target file selection
5. approval check

### FR-021 Post-edit pipeline
After edits, the assistant shall perform:
1. formatting if configured
2. lint/test/policy execution if configured
3. diff summarization
4. checkpoint update
5. user-visible status reporting

### FR-022 Narrow edit scope
The assistant shall distinguish between:
- editable files
- reference-only files
- excluded files

### FR-023 Diff-first UX
The UI should show diffs before final acceptance for destructive or non-trivial modifications, unless full-auto mode is explicitly enabled.

## 8.6 Git command execution policy

### FR-024 Command classification
Git commands shall be classified into risk tiers:

**Tier 0 — read-only**  
`git status`, `git diff`, `git log`, `git branch --show-current`, `git rev-parse`

**Tier 1 — reversible local write**  
checkpoint creation, branch creation, local commit in approved modes

**Tier 2 — potentially disruptive local write**  
`git reset`, `git checkout --`, stash operations affecting uncommitted work, rebase, amend, clean

**Tier 3 — remote side effects**  
`git push`, branch deletion on remote, PR creation, review submission, merge operations

### FR-025 Approval policy mapping
Approval requirements shall be configurable by tier.

Recommended default:
- Tier 0: no approval
- Tier 1: allowed in workspace if enabled by policy
- Tier 2: explicit approval required
- Tier 3: explicit approval always required

### FR-026 Sandboxed behavior
If `.git` or the repository metadata is read-only in the execution environment, the assistant shall degrade gracefully and explain which Git actions are unavailable.

## 8.7 Branch management

### FR-027 Branch naming assistance
The assistant shall be able to propose branch names derived from issue IDs, task summaries, or repository branch conventions.

### FR-028 Base branch inference
The assistant should infer a likely base branch using repository defaults, remote HEAD metadata, user preference, or prior task context.

### FR-029 Branch safety checks
Before branch creation or switching, the assistant shall verify:
- uncommitted changes policy
- conflict state
- permission to write refs

### FR-030 Branch metadata in artifacts
Generated commits and PR artifacts should include branch context when relevant.

Example naming policy:

```ts
export function suggestBranchName(input: {
  issueKey?: string;
  taskSummary: string;
  type?: "feature" | "bugfix" | "chore" | "docs";
}): string {
  const slug = input.taskSummary
    .toLowerCase()
    .replace(/[^a-z0-9]+/g, "-")
    .replace(/^-|-$/g, "")
    .slice(0, 48);

  return [input.type ?? "task", input.issueKey, slug].filter(Boolean).join("/");
}
```

## 8.8 Commit generation

### FR-031 Commit boundary discipline
The assistant shall group edits into logically coherent commit units.

### FR-032 Commit message generation
The assistant shall generate commit messages from the actual diff, not only from the user prompt.

### FR-033 Commit style profiles
The system shall support configurable commit conventions, including:
- free-form descriptive style
- Conventional Commits
- organization-specific templates

### FR-034 Commit message validation
If commit linting is configured, the assistant shall validate generated messages before commit.

### FR-035 Human override
Users shall be able to edit or replace generated commit messages before final commit.

### FR-036 Avoid mixing validation-only and feature changes
Where possible, the assistant shall separate pure formatting or generated-file updates from substantive logic changes.

Example:

```ts
export interface CommitProposal {
  title: string;
  body?: string;
  conventionalType?: "feat" | "fix" | "docs" | "refactor" | "test" | "chore";
  scopes?: string[];
  linkedIssues?: string[];
  rationale: string[];
}
```

Example generator stub:

```ts
export function buildCommitProposal(diffSummary: string, style: "conventional" | "plain"): CommitProposal {
  if (style === "conventional") {
    return {
      title: `fix: align implementation with repository validation rules`,
      body: `- updates target files only\n- preserves existing user changes\n- passes configured validation pipeline`,
      conventionalType: "fix",
      rationale: [diffSummary]
    };
  }

  return {
    title: "Align implementation with repository validation rules",
    body: diffSummary,
    rationale: [diffSummary]
  };
}
```

## 8.9 Pull request artifact generation

### FR-037 PR artifact support
The assistant shall generate:
- PR title
- PR description
- testing/validation section
- risk section
- linked issues or tickets
- rollout or migration notes if applicable

### FR-038 Diff-grounded PR summaries
PR artifacts shall be generated from actual changed files, commit summaries, and validation results.

### FR-039 Template support
The assistant shall support repository PR templates and organization-specific sections.

### FR-040 Base-branch-aware instructions
When generating PR review artifacts, the assistant shall resolve repository instructions from the base branch when the host platform’s review semantics depend on it.

### FR-041 PR-size awareness
If a PR exceeds configurable thresholds for files changed, lines changed, or directories touched, the assistant shall warn that the PR may be too large and should be split.

Example PR body skeleton:

```md
## Summary
- Adds shadow-checkpoint support for reversible AI edits
- Introduces diff-aware commit and PR artifact generation
- Integrates repository/path-scoped assistant instructions

## Why
The previous workflow mixed user edits and assistant edits, which increased rollback risk and reduced review clarity.

## Validation
- npm run lint
- npm test
- targeted integration tests for git checkpoint restore

## Risks
- repositories with custom hooks may require command allow-list updates
- shadow repository storage must be excluded from workspace search

## Linked items
- Closes ABC-123
```

## 8.10 GitHub integration

### FR-042 GitHub remote detection
The assistant shall detect whether a GitHub remote exists and normalize repository owner/name metadata.

### FR-043 GitHub capability discovery
The assistant shall know whether it can:
- read PR metadata
- create PRs
- update PR bodies
- read issues
- link issues
- post review comments
- read code review comments
- read check-run status

### FR-044 PR lifecycle support
Where credentials and permissions exist, the assistant shall support:
- draft PR creation
- PR artifact refresh
- linking commits to issues
- reading existing review comments for revision tasks

### FR-045 Check-run/status ingestion
The assistant should ingest GitHub check-run or status results as structured task context.

### FR-046 Issue-aware branch and PR generation
The assistant should infer and propagate issue IDs across branch naming, commit proposals, and PR artifacts.

### FR-047 Review-feedback loop
The assistant shall be able to consume GitHub PR review feedback, map comments to files/hunks, and plan targeted remediation.

### FR-048 Non-destructive remote policy
Creation, update, merge, and review-submission operations shall always obey explicit approval and repository permission policy.

## 8.11 Repository instructions and policy files

### FR-049 Repository-wide instructions
The assistant shall support repository-wide instruction files.

### FR-050 Path-specific instructions
The assistant shall support path-specific instruction files with glob or path matching.

### FR-051 Layered precedence
Instruction precedence shall be deterministic.

Recommended precedence order:
1. system policy
2. organization policy
3. repository-wide instructions
4. path-specific instructions
5. task-specific temporary instructions
6. user prompt

### FR-052 Character and token budget control
The instruction loader shall enforce size budgets and produce warnings when files exceed practical limits.

### FR-053 Validation-specific guidance
Instructions may include build, test, lint, formatting, commit, PR, security, migration, and review guidance.

Recommended file support:
- `AGENTS.md`
- `.github/copilot-instructions.md`
- `.github/instructions/**/*.instructions.md`
- `.claude/settings.json` or equivalent hook config references
- `.assistant/rules/**/*.md`
- repository PR templates and commit templates

Example normalized instruction resolution result:

```ts
export interface ResolvedInstructions {
  repositoryWide: Array<{ path: string; content: string }>;
  pathSpecific: Array<{ path: string; appliesTo: string[]; content: string }>;
  mergedText: string;
  truncated: boolean;
}
```

## 8.12 Validation and repair loop

### FR-054 Validation commands
The assistant shall support repository-configured commands for:
- formatter
- linter
- unit tests
- integration tests
- type checks
- custom policy hooks

### FR-055 Progressive validation
Validation shall be progressive where possible:
1. format
2. type/lint
3. targeted tests
4. broader suite if needed

### FR-056 Structured validation output
Validation results shall be parsed into structured diagnostics and summarized compactly for the model.

### FR-057 Repair loop
If configured, the assistant may attempt targeted repair based on validation failures, bounded by retry limits and approval policy.

### FR-058 Validation provenance
PR and commit artifacts shall reference the exact validation commands and outcomes used.

Example diagnostic contract:

```ts
export interface ValidationDiagnostic {
  source: "eslint" | "tsc" | "pytest" | "custom";
  severity: "error" | "warning" | "info";
  file?: string;
  line?: number;
  column?: number;
  code?: string;
  message: string;
}
```

- risky changes
- missing tests
- migration concerns

### FR-060 Review comment grounding
All generated review observations should reference concrete files/hunks where possible.

- draft-ready

## 8.14 Memory, plan, and task artifacts

### FR-063 Separate memory from Git history
Project memory and task memory shall be stored independently from commit history.

### FR-064 Repository-shared memory scope
Memory may be shared across worktrees or subdirectories of the same repository when configured.

### FR-065 Task plan artifact
The assistant shall maintain a structured plan artifact for long-running tasks.

The assistant shall support resuming a prior task using a combination of plan state, checkpoints, and repository state.

## 9. Non-functional requirements

## 9.1 Safety and data integrity

### NFR-001 No silent destructive Git actions
The assistant shall not perform destructive Git actions silently.

### NFR-002 Reversibility
All assistant-triggered file modifications shall be reversible through checkpoint restore or explicit Git history.

### NFR-003 Dirty-worktree preservation
User-authored uncommitted work shall not be lost due to assistant actions.

### NFR-004 Auditability
Every Git or GitHub side effect shall be logged with actor, action, timestamp, repository, branch, and outcome.

---

## 9.2 Performance

### NFR-005 Status and diff freshness
Repository status and diff context shall be refreshed before major edit, commit, or PR actions.

### NFR-006 Token efficiency
Prompt assembly shall prefer branch/diff summaries and structured metadata over raw file blobs whenever possible.

### NFR-007 Bounded validation loops
Automated repair loops shall obey configurable retry and time budgets.

---

## 9.3 Usability

### NFR-008 Predictable approvals
Approval prompts shall explain exactly what side effect will occur.

### NFR-009 Clear rollback affordance
The UI shall expose an easy path to restore the previous assistant checkpoint.

### NFR-010 Review clarity
Diffs, commit proposals, and PR artifacts shall be understandable without reading the full chat transcript.

---

## 9.4 Enterprise readiness

### NFR-011 Policy-driven configuration
Organizations shall be able to disable or restrict branch creation, remote push, PR creation, or visible auto-commit behavior.

### NFR-012 Offline-degraded operation
The assistant shall still support local Git workflows when GitHub access is unavailable.

### NFR-013 Secret-conscious logging
Logs and telemetry shall avoid storing secrets, token values, or file contents unless explicitly configured.

## 10. Recommended normalized behavior model

This section resolves differences across source products into a single coherent recommendation.

### 10.1 Default checkpoint strategy
**Recommendation:** use a shadow Git checkpoint store by default.

Rationale:
- keeps user-visible history clean
- allows near-zero-cost experimentation
- avoids cluttering the main branch with machine-generated commits
- still provides robust restore semantics

### 10.2 Optional visible commit mode
Offer visible main-repo auto-commit mode for advanced users or terminal-centric workflows that prefer native Git-only undo.

### 10.3 Default approval strategy
- read-only Git inspection: auto-allow
- local checkpoint creation: auto-allow
- visible commits in main repo: ask on first use, then remember if allowed
- stash/reset/rebase/amend/clean: explicit approval every time unless organization policy overrides
- remote push / PR creation / merge: explicit approval every time

### 10.4 Default diff usage
Use Git diff as the primary context for:
- commit messages
- PR descriptions
- pre-commit review
- issue linkage suggestions
- regression risk summaries

### 10.5 Default instruction strategy
Use layered instructions with strict precedence and size caps. Resolve repository-wide guidance plus path-specific guidance that matches touched files.

### 10.6 Default validation strategy
Run formatting and targeted validation automatically after edits where configured. Offer broader suite execution before PR creation or upon request.

## 12. Workflow specifications

## 12.1 Edit task workflow

```text
1. Detect repo and capabilities
2. Refresh status and instruction layers
3. If dirty, preserve pre-existing changes according to policy
4. Create checkpoint
5. Select target files and reference context
6. Apply edits
7. Show diff / summarize changes
8. Run format + validation
9. Optionally repair failures within retry budget
10. Produce commit proposal and completion summary
```

## 12.2 Commit workflow

```text
1. Refresh status
2. Confirm eligible changed files
3. Generate diff-grounded commit proposal
4. Validate message against policy
5. Request approval if visible commit required
6. Commit
7. Record audit event and update task plan
```

## 12.3 Pull request workflow

```text
1. Detect GitHub remote and current branch
2. Infer base branch
3. Gather diff, commit summaries, validation results, linked issues
4. Resolve PR template and repository instructions
5. Generate PR title/body
6. Ask for approval
7. Create or update PR draft
8. Store PR metadata in task artifacts
```

## 12.4 Review remediation workflow

```text
1. Ingest PR review comments and check-run failures
2. Map feedback to files/hunks/tests
3. Build targeted plan
4. Create checkpoint
5. Apply fixes
6. Run targeted validation
7. Update commit/PR artifacts
```

## 13. Prompt and context recommendations

## 13.1 What Git context should be included
For Git-aware tasks, prompt assembly should include compact structured context such as:

```json
{
  "repo": {
    "root": "/workspace/project",
    "head": "feature/abc-123-shadow-checkpoints",
    "base": "main",
    "state": "dirty-assistant-only"
  },
  "diff": {
    "changedFiles": [
      "src/git/checkpoints.ts",
      "src/ui/checkpointPanel.tsx"
    ],
    "summary": "Adds checkpoint creation and restore UI, updates task orchestration to checkpoint before edits"
  },
  "validation": {
    "commands": ["npm run lint", "npm test -- checkpoints"],
    "status": "failing",
    "diagnostics": 2
  }
}
```

### Recommendation
For commit and PR tasks, do not send entire edited files unless needed. Prefer:
- branch name
- changed path list
- diff summary
- validation summary
- linked issue IDs
- repository instructions

## 13.2 Instruction merging algorithm

```ts
export function mergeInstructionLayers(layers: string[]): string {
  return layers
    .map(x => x.trim())
    .filter(Boolean)
    .join("\n\n---\n\n");
}
```

### Requirement
The merged result shall preserve source metadata separately even if the final model input is flattened.

## 13.3 Diff summarization before commit/PR generation

```ts
export interface DiffSummary {
  filesChanged: number;
  additions: number;
  deletions: number;
  majorAreas: string[];
  behavioralChanges: string[];
  validationImpact: string[];
}
```

Use the summary object, not raw `git diff`, for first-pass artifact generation. Fall back to hunks only when higher fidelity is needed.

## 14. Safety controls

## 14.1 High-risk operation guardrails
The assistant shall never perform the following without explicit approval unless a stricter enterprise automation policy intentionally allows it:
- `git reset --hard`
- `git clean -fd`
- `git rebase`
- `git push --force`
- remote branch deletion
- merge to protected branches
- PR merge

## 14.2 Protected branches
The assistant shall recognize protected-branch patterns and downgrade capabilities accordingly.

## 14.3 Secret and credential awareness
GitHub tokens, SSH keys, and credential helper outputs shall never be echoed into model prompts or logs.

## 14.4 Hook safety
Repository-defined hooks or assistant lifecycle hooks shall execute only under configured policy and with bounded timeouts.

## 16. GitHub review and status-check strategy

### Recommendation
Treat GitHub as the canonical collaboration surface, but not the canonical working-memory surface.

The assistant should:
- read check-run outcomes as structured signals
- generate PR descriptions using repository templates and validation outcomes
- consume review comments and map them to local code context
- optionally publish AI-generated PR summaries or status outputs where permitted

### Recommended best practices
1. Generate PR content from actual diff and validation output.
2. Respect repository instructions from the base branch when creating review-targeted artifacts.
3. Keep review comments precise and file-grounded.
4. Avoid posting low-confidence comments automatically.
5. Separate local checkpointing from remote review publication.

## 19. Acceptance criteria

The enhancement shall be accepted only if all of the following are satisfied.

### AC-001 Dirty-tree protection
Given a dirty repository with pre-existing user changes, when the assistant applies edits, the pre-existing changes remain recoverable and distinct from AI changes.

### AC-002 Restore reliability
Given any assistant checkpoint created during a task, restoring it returns the workspace to the recorded state without losing task memory.

### AC-003 Diff-grounded commit messages
Generated commit messages reflect the actual diff and pass configured commit policy checks.

### AC-004 PR artifact completeness
Generated PR artifacts include summary, rationale, validation, and risk sections when applicable.

### AC-005 Validation loop correctness
When validation fails after an assistant edit, the assistant can surface structured diagnostics and optionally perform bounded repair.

### AC-006 Approval correctness
Tier-2 and Tier-3 Git/GitHub actions do not execute without the configured approval path.

### AC-007 Path-specific instructions
When touched files match path-specific instruction rules, those rules are incorporated into the task context.

### AC-008 Offline degradation
When GitHub is unavailable, local Git workflows remain functional and the assistant explains which remote capabilities are unavailable.

## 24. Example end-to-end pseudo-code

```ts
async function executeEditTask(task: string, cwd: string) {
  const repo = await git.discoverRepo(cwd);
  if (!repo.isGitRepo) throw new Error("No Git repository found");

  const status = await git.getStatus(repo.rootPath);
  const instructions = await instructionResolver.resolve({
    repoRoot: repo.rootPath,
    candidatePaths: status.changedPaths.map(x => x.path)
  });

  if (status.state === "conflicted") {
    return {
      mode: "conflict-resolution",
      message: "Repository has merge conflicts. Resolve conflicts before unrelated edits."
    };
  }

  const checkpoint = await git.createCheckpoint({
    repoRoot: repo.rootPath,
    label: `Before task: ${task}`,
    mode: "shadow"
  });

  const context = await promptBuilder.build({
    task,
    repo,
    status,
    instructions,
    checkpoint
  });

  const editResult = await model.applyEdits(context);
  await formatter.run(editResult.changedFiles);

  const validation = await validator.run(repo.rootPath, editResult.changedFiles);
  if (!validation.ok) {
    await repairLoop.tryFix(validation.diagnostics, { maxAttempts: 2 });
  }

  const diff = await git.getDiff(repo.rootPath);
  const commitProposal = buildCommitProposal(diff, "conventional");

  return {
    checkpoint,
    changedFiles: editResult.changedFiles,
    validation,
    commitProposal
  };
}
```

## 25. Explicit design recommendations to implementation team

1. **Adopt shadow checkpoints as the default safety primitive.**  
   This best reconciles terminal-first Git-native tools with IDE tools that emphasize low-friction restore.

2. **Use Git diff as a first-class context provider.**  
   Commit messages, PR descriptions, and review summaries should be diff-grounded, not prompt-grounded.

3. **Do not rely on raw conversation history for task continuity.**  
   Persist plan, memory, and checkpoint metadata separately.

4. **Implement instruction layering once and reuse it across edit, review, commit, and PR flows.**

5. **Keep destructive Git operations behind hard approvals.**

6. **Design the GitHub integration as an optional capability layer.**  
   Local Git workflows must remain fully usable without GitHub connectivity.

7. **Make rollback highly visible in the UI.**  
   Users trust aggressive agents only when restore is obvious and reliable.

8. **Bound autonomous repair loops.**  
   Prefer at most 2–3 repair cycles before surfacing control back to the user.

9. **Treat repository templates and policy files as executable guidance.**  
   The assistant should resolve and use them automatically rather than expecting users to paste them into prompts.

10. **Instrument everything.**  
    Adoption will depend on measurable reduction in broken changes, review churn, and revert pain.

## 26. Summary

A high-quality Git and GitHub integration for a coding assistant is not merely a thin wrapper around `git status`, `git commit`, and PR creation. The best observed systems converge on a deeper pattern:

- Git provides reversible state and diff-aware truth.
- GitHub provides collaboration, review, and automation surfaces.
- Repository instruction files provide durable local policy.
- Checkpoints make autonomous editing safe.
- Validation and review loops turn edits into mergeable work.
- Human approval remains the boundary for high-impact actions.

This specification translates those patterns into a coherent implementation model suitable for an existing assistant and ready for engineering execution.

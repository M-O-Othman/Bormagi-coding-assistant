# AI Coding Assistant Feature Specification
## Feature: Chatbot AI / UX

**Document Type:** Implementation-ready requirements specification with embedded design recommendations  
**Target Audience:** Product owners, solution architects, frontend engineers, backend engineers, agent/runtime engineers, QA, security reviewers  
**Scope:** Chat-first user experience for an existing coding assistant across IDE, terminal, and web-style surfaces  
**Status:** Draft for implementation  
**Version:** 1.0

---

## 1. Purpose

This document specifies the target **chatbot AI / UX** capabilities for an existing coding assistant. It consolidates implementation patterns and design lessons inspired by:

- OpenAI Codex
- Claude Code
- GitHub Copilot
- Aider
- Cline
- Continue
- Roo Code

The document intentionally **normalizes** those ideas into a single coherent design. It does **not** attempt to copy any single product. Where source products differ, this specification resolves the differences into a consistent product direction suitable for an implementation team.

This specification covers:

- conversational UX behavior
- task lifecycle and mode system
- context visibility and trust
- approval and autonomy UX
- plan-first and edit-first interaction models
- artifacts and memory in chat
- tool call presentation
- recovery, checkpoints, and resumability
- onboarding and discoverability
- accessibility and responsiveness
- extensibility for custom commands, agents, and skills

This specification does **not** define the low-level retrieval engine, sandbox internals, or Git/GitHub integration in depth. Those are assumed to exist or be specified in companion documents.

---

## 2. Product Goal

The assistant SHALL provide a **professional, transparent, and low-cognitive-load conversational interface** for software engineering work.

The UX SHALL enable users to:

1. ask questions about code safely without accidental edits,
2. plan changes before acting,
3. execute multi-step coding tasks with visible progress,
4. understand what context, files, tools, and permissions are in play,
5. review proposed changes before risky operations,
6. recover from bad actions quickly,
7. resume long-running or interrupted work coherently,
8. customize behavior at team, repo, path, and task level.

The design SHALL optimize for:

- trust,
- reversibility,
- explainability,
- fast task completion,
- low token waste,
- minimal unnecessary UI complexity.

## 3. Design Principles

### 3.1 Plan before action
The UX SHALL default to a model where the user can explore, ask, inspect, and plan before changes are made. Read-only planning must feel first-class rather than secondary.

### 3.2 Show the operating state clearly
At all times, the user SHALL be able to determine:

- current mode,
- allowed actions,
- active workspace/repository,
- selected files and relevant context,
- pending approvals,
- whether the assistant is autonomous or paused,
- whether the assistant is reading, editing, running tools, or waiting.

### 3.3 Make side effects visible and reviewable
The assistant SHALL never hide meaningful side effects. Any file change, command execution, or network action must be represented in the UI as a visible action with result state.

### 3.4 Support progressive disclosure
The default UI SHALL remain simple. Advanced controls such as detailed token diagnostics, context internals, hook traces, and tool routing SHALL be available on demand without overloading basic interactions.

### 3.5 Keep chat grounded in artifacts
Long engineering work SHALL not rely only on prose chat history. The UX SHALL elevate structured artifacts such as:

- plans,
- task lists,
- diffs,
- test results,
- checkpoints,
- selected files,
- rule sources,
- reusable commands.

### 3.6 Preserve user agency
Automation SHALL accelerate work, not obscure control. Users must be able to pause, approve, reject, edit the plan, revert changes, and constrain future actions.

### 3.7 Keep interaction costs low
The UX SHALL reduce the need for repeated prompting by exposing reusable modes, commands, instructions, path-scoped rules, and skills.

### 3.8 Prefer clarity over anthropomorphism
The assistant SHALL present itself as a professional engineering tool, not a toy persona. Avoid childish visuals, decorative avatars, or emotional framing that weakens trust.

## 4. Target User Jobs

The UX must support at least the following user jobs:

1. **Ask**: Explain code, summarize architecture, answer questions, inspect logs, compare options.
2. **Plan**: Produce implementation plan, migration plan, debugging plan, review checklist.
3. **Edit**: Make code changes in a controlled scope.
4. **Debug**: Investigate failures through logs, stack traces, tests, and command outputs.
5. **Review**: Review diffs, explain changes, identify risks, generate comments.
6. **Run**: Execute allowed commands such as tests, lint, builds, or project scripts.
7. **Refine**: Continue a prior task, revise a plan, or re-run after feedback.
8. **Delegate**: Invoke a skill, custom command, or specialized agent/profile.
9. **Recover**: Undo changes, restore checkpoint, continue interrupted work.
10. **Learn**: Discover how to use the assistant and what it can do in the current environment.

## 5. Interaction Model

### 5.1 Primary interaction style
The primary interaction style SHALL be **chat-centered task orchestration** with artifact-backed execution.

This means:

- the user communicates primarily through natural language,
- the assistant answers in chat,
- but the state of work is represented with structured panels/cards,
- not buried only in natural-language responses.

### 5.2 Required conversation phases
Every substantial task SHALL progress through these conceptual phases, even if the UI compresses them:

1. **Intent capture** – understand task and scope.
2. **Context assembly** – identify files, symbols, diagnostics, instructions, and tools.
3. **Plan or direct action decision** – determine whether to stay read-only or act.
4. **Execution** – edits, commands, retrieval, tool calls.
5. **Verification** – tests, lint, runtime checks, or explicit non-verification note.
6. **Review summary** – explain outcomes and remaining risks.
7. **Continuation state** – preserve task memory, plan state, and checkpoint metadata.

### 5.3 Message classes
The chat system SHALL support the following message classes:

- **User message**
- **Assistant narrative message**
- **Plan artifact**
- **Action log entry**
- **Tool execution result**
- **Approval request**
- **Diff/change summary**
- **Warning/risk notice**
- **Checkpoint/resume notice**
- **System state/status message**

The UI SHOULD render these as visually distinct blocks.

## 6. Modes and UX States

The assistant SHALL support explicit operating modes. These modes unify ideas common across multiple source systems while avoiding provider-specific coupling.

### 6.1 Required built-in modes

#### 6.1.1 Ask Mode
Purpose: read-only Q&A and explanation.

Allowed actions:
- read files,
- search code,
- inspect symbols,
- inspect logs and diffs,
- summarize repository knowledge,
- answer conceptual questions.

Disallowed by default:
- file edits,
- command execution,
- network side effects,
- git mutation.

UX requirements:
- visibly labeled “Ask” or equivalent,
- explicit read-only badge,
- answer should cite which files/symbols informed the response when applicable.

#### 6.1.2 Plan Mode
Purpose: convert a request into a high-quality execution plan before edits.

Allowed actions:
- all Ask Mode actions,
- read-only inspection of files and structure,
- plan generation,
- task decomposition,
- risk identification,
- identifying missing inputs.

Disallowed by default:
- file edits,
- command execution,
- state mutation.

UX requirements:
- plan artifact generated as structured checklist,
- user can approve full plan or approve step-by-step,
- user can edit/reorder/disable steps.

#### 6.1.3 Code Mode
Purpose: implement changes.

Allowed actions:
- propose file changes,
- apply edits,
- read supporting files,
- run pre-approved safe commands,
- produce diffs and summaries.

UX requirements:
- selected editable scope visible,
- every edit grouped by file,
- if a large change touches many files, the user sees a grouped change summary before commit-like finalization.

#### 6.1.4 Debug Mode
Purpose: isolate and fix issues.

Allowed actions:
- inspect stack traces,
- run tests,
- grep/search code,
- add temporary diagnostics if allowed,
- propose and apply fixes,
- compare hypotheses.

UX requirements:
- hypotheses list,
- evidence links to logs/tests/files,
- verification result after each attempted fix.

#### 6.1.5 Review Mode
Purpose: inspect diffs, summarize changes, identify risk and follow-up work.

Allowed actions:
- read diffs,
- inspect modified files,
- inspect tests,
- generate review comments,
- create risk register.

UX requirements:
- review summary structured into correctness, maintainability, security, performance, and test coverage where applicable.

### 6.2 Custom modes
The system SHALL support user-defined or team-defined modes.

Custom modes SHALL allow configuration of:
- display name,
- description,
- allowed tools,
- default instruction layers,
- default context providers,
- approval profile,
- preferred model role,
- default output schema.

### 6.3 Mode transitions
The UI SHALL support:
- explicit mode switching by dropdown or command,
- inline mode switch inside a conversation,
- sticky mode for future messages,
- suggested mode switch when the user intent conflicts with the current mode.

Example:
- if user asks “change these files” in Ask Mode, the assistant SHOULD suggest transition to Code Mode rather than silently editing.

## 7. Core Chat UX Requirements

### 7.1 Composer requirements
The chat composer SHALL support:
- multiline text,
- code block pasting,
- drag/drop files or snippets,
- slash commands,
- @-style context mentions,
- mode indicator,
- send shortcut and configurable alternate send behavior.

The composer SHOULD support:
- prompt templates,
- prompt history,
- pinned reusable task starters,
- selection-aware prompts from the IDE/editor.

### 7.2 Slash commands
The assistant SHALL support slash commands for discoverable structured actions.

Minimum commands:
- `/ask`
- `/plan`
- `/code`
- `/debug`
- `/review`
- `/clear`
- `/resume`
- `/checkpoint`
- `/help`

The system SHOULD also support:
- `/test`
- `/lint`
- `/explain`
- `/fix`
- `/spec`
- `/skill <name>`
- `/agent <name>`

Slash commands SHALL:
- be searchable,
- show descriptions and examples,
- expose expected arguments,
- optionally pre-fill structured prompts.

### 7.3 Context mentions
The composer SHALL support mention-based context injection.

Mentionable entities SHOULD include:
- files,
- folders,
- symbols,
- diffs,
- terminal output,
- test failures,
- problems/diagnostics,
- documentation sources,
- tasks/plans,
- checkpoints,
- custom knowledge sources.

Example UX:

```text
@src/payments/InvoiceService.ts explain how retries are handled
@test:latest-failures fix the broken cases
@diff review this change for backward compatibility risks
@plan continue from step 3
```

### 7.4 Smart suggestions
The UI SHOULD suggest context candidates while composing.

Examples:
- if the user pastes a stack trace, suggest failed files and tests,
- if the user references a symbol, suggest matching definitions,
- if the user asks “continue”, suggest last unfinished task or checkpoint,
- if the user asks “why did this fail?”, suggest latest terminal output.

### 7.5 Response formatting
Assistant responses SHALL be formatted for engineering work, not generic consumer chat.

Responses SHOULD prefer:
- short conclusion first,
- structured sections,
- explicit assumptions,
- clear file and symbol references,
- checklists where action is needed,
- concise explanations followed by expandable details.

The assistant SHALL avoid:
- overly verbose motivational language,
- anthropomorphic filler,
- hidden assumptions,
- large uninterrupted text blocks when artifacts or tables would be clearer.

## 8. Artifact-Centered UX

Long-running engineering conversations degrade when everything is treated as normal chat. The system SHALL elevate key outputs into artifacts.

### 8.1 Required artifact types
- **Plan artifact**
- **Task checklist**
- **Diff summary artifact**
- **Verification artifact**
- **Risk register**
- **Checkpoint artifact**
- **Instruction/rule source summary**
- **Resume state artifact**

### 8.2 Plan artifact schema
A plan artifact SHALL contain:
- task title,
- objective,
- assumptions,
- prerequisites,
- ordered steps,
- optional dependencies between steps,
- risk notes,
- validation approach,
- completion status per step.

Example:

```ts
export interface PlanArtifact {
  id: string;
  title: string;
  objective: string;
  assumptions: string[];
  prerequisites: string[];
  steps: Array<{
    id: string;
    title: string;
    description: string;
    status: "todo" | "in_progress" | "done" | "blocked" | "skipped";
    dependsOn?: string[];
    validation?: string[];
    riskNotes?: string[];
  }>;
  createdAt: string;
  updatedAt: string;
}
```

### 8.3 Diff summary artifact
The assistant SHALL generate a change summary artifact after any non-trivial edit.

It SHALL include:
- changed files,
- high-level intent of each change,
- possible side effects,
- validation performed,
- remaining risks,
- revert/checkpoint reference.

### 8.4 Persistent task artifact
For multi-turn work, the system SHALL maintain a task artifact independent of raw chat history.

This artifact SHALL preserve:
- current objective,
- current mode,
- active plan,
- selected files,
- unresolved blockers,
- last successful validation,
- latest checkpoint reference.

## 9. Transparency and Trust UX

### 9.1 Visible context panel
The assistant SHALL expose a “Current Context” panel or equivalent view.

It SHALL show:
- active repo/workspace,
- mode,
- selected files,
- referenced symbols,
- active instructions/rules,
- tool permissions in effect,
- current checkpoint,
- token/context budget summary.

The panel SHOULD support expanding each item to explain **why it is included**.

Example structure:

```ts
export interface ContextItemViewModel {
  id: string;
  type: "file" | "symbol" | "diff" | "terminal" | "instruction" | "memory" | "doc";
  label: string;
  source: string;
  reasonIncluded: string;
  estimatedTokens?: number;
  removable: boolean;
}
```

### 9.2 Action timeline
Every tool call or side effect SHALL appear in an action timeline.

Each entry SHALL include:
- action name,
- target,
- status,
- timestamp,
- short result summary,
- expandable raw details.

Example entries:
- Read file `src/api/auth.ts`
- Search symbol `InvoiceService`
- Run command `npm test -- InvoiceService.spec.ts`
- Edit file `src/api/auth.ts`
- Create checkpoint `cp_20260308_001`

### 9.3 Explain-why affordance
For meaningful decisions, the assistant SHOULD offer an explanation affordance such as “Why this?” or “Why these files?”

It SHALL explain in bounded, compact form:
- why particular files were selected,
- why a command was run,
- why a permission is requested,
- why a mode switch is recommended.

### 9.4 Confidence and uncertainty
The assistant SHALL explicitly signal uncertainty when applicable.

This includes:
- missing context,
- ambiguous requirements,
- failing verification,
- partial repository visibility,
- instruction conflicts,
- plan assumptions.

The UX SHALL prefer statements like:
- “I found two plausible implementations.”
- “This plan assumes API v2 is the intended target.”
- “I could not verify runtime behavior because tests require unavailable secrets.”

## 10. Approval UX and Autonomy Controls

### 10.1 Approval levels
The product SHALL support at least three approval profiles:

1. **Strict** – approve every file change, command, or network action.
2. **Balanced** – auto-approve low-risk reads and safe commands; prompt for edits and risky commands.
3. **Autonomous** – allow pre-approved operations within the policy boundary; pause only for high-risk actions or blockers.

### 10.2 Approval prompts
Approval prompts SHALL be actionable and compact.

Each prompt SHALL state:
- requested action,
- why it is needed,
- affected scope,
- risk level,
- alternatives if denied.

Example:

```text
Approval needed: Run `npm test -- billing`.
Reason: Validate the proposed retry fix in the billing module.
Scope: Local test process only, no network.
Risk: Low.
```

### 10.3 Batch approvals
The user SHOULD be able to approve:
- this action once,
- this class of action for this task,
- this class of action for this repo/session,
- deny always for this task/session.

### 10.4 Live autonomy status
The UI SHALL display whether the assistant is:
- waiting for input,
- waiting for approval,
- executing autonomously,
- paused by policy,
- blocked by error.

### 10.5 Guarded autonomy pattern
When autonomy is enabled, the assistant SHALL still create internal milestones and pause at meaningful boundaries such as:
- after plan generation,
- after large grouped edits,
- after validation failure,
- before external side effects,
- when multiple approaches are possible.

## 11. Checkpoints, Undo, and Recovery UX

### 11.1 Checkpoint visibility
The assistant SHALL support checkpoints as a first-class concept.

The UI SHALL show:
- latest checkpoint,
- checkpoint history for the task/session,
- concise description of what changed since each checkpoint.

### 11.2 Restore flow
Users SHALL be able to restore to a checkpoint through the UI.

The restore flow SHALL:
- show impacted files,
- explain what will be lost,
- confirm restoration.

### 11.3 Automatic checkpoint policy
The system SHOULD auto-create checkpoints:
- before the first edit in a task,
- before multi-file edits,
- before running refactors,
- before high-risk commands,
- before switching from plan to code mode.

### 11.4 Resume flow
The UI SHALL support resuming previous work.

Resume SHALL reconstruct:
- task objective,
- last mode,
- active plan,
- pending approvals,
- open blockers,
- last known validation state,
- relevant diffs/checkpoints.

Example resume state:

```ts
export interface ResumeState {
  taskId: string;
  title: string;
  mode: "ask" | "plan" | "code" | "debug" | "review";
  lastSummary: string;
  nextSuggestedAction: string;
  selectedFiles: string[];
  checkpointId?: string;
  planId?: string;
  blockers: string[];
}
```

## 12. Multi-Step Task UX

### 12.1 Task board within chat
For multi-step work, the assistant SHALL render a lightweight task board or checklist inside the conversation.

Each step SHALL show:
- state,
- owner (assistant/user),
- linked files or artifacts,
- validation result,
- notes.

### 12.2 Long-horizon execution
The assistant SHALL support long-horizon tasks without requiring the user to remember the full prior chat.

This SHALL be achieved through:
- persistent task artifact,
- concise rolling summaries,
- plan step status,
- checkpoint references,
- reusable instruction layers.

### 12.3 Human interruption tolerance
If the user interrupts with a new request, the assistant SHALL:
- preserve current task state,
- allow side-thread question answering,
- provide a clean “return to previous task” affordance.

## 13. Context UX and Cognitive Load Management

### 13.1 Context budget visibility
The UI SHOULD expose a simplified context health indicator.

Suggested levels:
- Healthy
- Busy
- Near limit
- Compacted

### 13.2 Compaction UX
If the session is compacted or summarized, the user SHALL be notified.

The notification SHALL explain:
- that older chat was condensed,
- what was preserved,
- what was dropped or demoted,
- how to inspect the summary artifact.

### 13.3 What gets preserved
The compaction UX SHALL emphasize that durable items are preserved preferentially:
- plans,
- decisions,
- checkpoints,
- diffs,
- validation results,
- selected files,
- rule sources.

### 13.4 What gets dropped first
The compaction logic SHOULD demote:
- conversational filler,
- duplicate explanations,
- superseded hypotheses,
- obsolete command output.

### 13.5 Context pruning controls
Advanced users SHOULD be able to remove context items manually from the current task.

Examples:
- remove a noisy log attachment,
- remove an irrelevant file,
- pin a critical design document,
- mark certain instructions as task-irrelevant.

## 14. Tool Use UX

### 14.1 Tool taxonomy
Tools SHALL be categorized in the UI at least as:
- Read
- Edit
- Search
- Run command
- Fetch docs
- Git/GitHub
- External integrations

### 14.2 Tool call rendering
Every tool call SHALL render as a compact card with:
- tool name,
- purpose,
- target,
- status,
- duration,
- result summary,
- expandable raw payload/log.

### 14.3 Tool failure UX
When a tool fails, the assistant SHALL not merely dump raw error text.

It SHALL provide:
- what failed,
- likely reason,
- whether the task can continue,
- next best action.

### 14.4 Safe defaults for command execution
Commands SHALL display with syntax-preserved formatting and clear scope.

Example:

```text
Running in workspace root:
npm test -- src/payments/__tests__/retry.spec.ts
```

### 14.5 Streaming output UX
For long-running commands, the UI SHOULD support streaming output with:
- live status,
- collapse/expand,
- terminal-like view,
- important line extraction.

## 15. Instruction, Rule, and Skill UX

### 15.1 Layered instructions
The UX SHALL represent instruction layers explicitly.

Supported layers SHOULD include:
- global user instructions,
- organization/team instructions,
- repository instructions,
- path-scoped instructions,
- mode-specific instructions,
- task-specific instructions.

### 15.2 Instruction conflict visibility
If instruction layers conflict, the assistant SHALL surface that conflict rather than silently behave inconsistently.

Example:
- repo instruction says “use Jest,” path instruction says “use Vitest.”
- UI should identify the higher-priority applicable instruction and show that a conflict existed.

### 15.3 Rule source viewer
The user SHOULD be able to inspect which rule files or instruction sources influenced the current reply or action.

### 15.4 Skills and agent profiles
The system SHALL support reusable skills/agent profiles/custom commands.

Each SHALL have:
- name,
- description,
- activation conditions,
- tools required,
- optional parameters,
- examples.

The chat UX SHALL make these discoverable through:
- slash command search,
- capability browser,
- context-sensitive suggestions.

## 16. Discoverability and Onboarding

### 16.1 First-run onboarding
The first-run experience SHALL teach the minimal mental model:
- Ask = read-only,
- Plan = design before changes,
- Code = edit and verify,
- approvals protect risky actions,
- checkpoints make edits reversible,
- @ mentions and slash commands improve specificity.

### 16.2 Empty-state examples
The empty state SHALL show concrete examples rather than generic marketing text.

Examples:
- “Explain how authentication works in this repo”
- “Plan a migration from Express to Fastify”
- “Fix the failing checkout tests”
- “Review this diff for security risks”

### 16.3 Contextual education
The UI SHOULD teach features at the moment they become relevant.

Examples:
- suggest Plan Mode when request is broad,
- suggest `@terminal` when build output exists,
- suggest checkpoint restore after repeated failures,
- suggest custom skill when user repeats a pattern.

## 17. Accessibility and Interaction Quality

### 17.1 Accessibility baseline
The interface SHALL meet modern accessibility standards for:
- keyboard navigation,
- focus order,
- screen reader labels,
- color contrast,
- resizable text,
- reduced motion support.

### 17.2 Keyboard-first workflow
A coding assistant MUST work efficiently without mouse dependence.

Minimum keyboard actions:
- open chat,
- switch mode,
- search commands,
- navigate message blocks,
- approve/deny action,
- open diff,
- restore checkpoint,
- insert context mention.

### 17.3 Compact vs expanded density
The UI SHOULD support at least two densities:
- compact for expert users,
- comfortable for default readability.

### 17.4 Responsive layout
The layout SHALL adapt to:
- narrow side panel,
- full-width pane,
- terminal/CLI conversational view,
- browser-based workspace.

## 18. Notification UX

### 18.1 Notification types
Notifications SHALL be limited to meaningful events:
- approval needed,
- task complete,
- validation failed,
- command blocked,
- context compacted,
- checkpoint created/restored.

### 18.2 Non-spam behavior
The system SHALL avoid excessive toast spam for every tool action. Routine reads should remain in the timeline rather than generating separate disruptive notifications.

### 18.3 Idle waiting notice
When waiting for user input or approval, the assistant SHOULD show a subtle but visible idle state notice.

### 19.1 Main chat pane
Contains:
- conversation timeline,
- plan artifacts,
- diff summaries,
- action cards,
- approvals.

### 19.2 Composer footer
Contains:
- mode selector,
- command/context hinting,
- send button,
- quick attach/context buttons.

### 19.3 Optional right rail / drawer
Contains:
- current context,
- task artifact,
- plan steps,
- selected files,
- rule sources,
- checkpoint list.

### 19.4 Optional bottom panel
Contains:
- streaming terminal output,

## 21. Error and Failure Handling UX

### 21.1 Clarify capability failures
When the assistant cannot do something because of policy, permissions, missing tools, or unavailable context, the UX SHALL state which class of constraint applies.

Examples:
- permission denied,
- tool unavailable,
- repo not indexed,
- network blocked,
- sandbox restriction,
- instruction conflict,
- validation environment missing.

### 21.2 Non-destructive failure recovery
After a failed action, the UI SHALL offer next-step options such as:
- retry,
- view raw output,
- switch mode,
- request approval,
- restore checkpoint,
- continue without that action.

### 21.3 Partial completion transparency
If part of a task completes, the assistant SHALL clearly separate:
- completed work,
- incomplete work,
- blocked work,
- recommended next step.

## 22. Personalization and Team UX

### 22.1 Personal preferences
The product SHOULD support user preferences for:
- preferred default mode,
- auto-open context panel,
- compact/comfortable density,
- response verbosity,
- whether to show tool internals by default,
- keyboard bindings.

### 22.2 Team-shared experiences
The product SHOULD support team-level reusable assets:
- command packs,
- custom modes,
- skills,
- instruction templates,
- review rubrics,
- debugging workflows.

### 22.3 Path-scoped specialization
The UX SHOULD reflect when different folders have different conventions.

Examples:
- frontend folder uses one test framework,
- backend folder uses another,
- infra folder requires extra review guardrails.

## 23. Non-Functional Requirements

### 23.1 Latency targets
The UI SHOULD meet these interaction targets under normal conditions:
- time to first streamed response token: <= 2.5 seconds for lightweight ask flows,
- plan artifact render: <= 5 seconds for moderate tasks,
- action card appearance after tool invocation: <= 500 ms,
- diff summary render after edit completion: <= 2 seconds.

### 23.2 Resilience targets
- no user-visible loss of task artifact during page refresh or IDE reload,
- recover active task state after restart where technically possible,
- preserve checkpoint metadata independently from transient chat stream.

### 23.3 Auditability
User-visible actions SHALL be reproducible from timeline and artifact data. The UI should allow exporting task summaries for engineering traceability.

### 23.4 Security and privacy
Sensitive outputs and secrets SHALL be redacted or clearly marked. The assistant SHALL avoid rendering secrets in plain text unless explicitly permitted by policy and necessary for the task.

## 24. Recommended State Model

```ts
export type AssistantMode = "ask" | "plan" | "code" | "debug" | "review" | string;

export interface ChatSessionState {
  sessionId: string;
  workspaceId: string;
  repoRoot?: string;
  mode: AssistantMode;
  autonomyProfile: "strict" | "balanced" | "autonomous";
  activeTask?: TaskState;
```

## 27. QA Acceptance Criteria

The following SHALL be testable acceptance criteria.

### 27.1 Ask mode safety
- In Ask Mode, asking code questions MUST NOT edit files.
- UI must clearly indicate read-only mode.

### 27.2 Plan artifact quality
- In Plan Mode, a request for a non-trivial change must produce a structured plan artifact.
- User can approve entire plan or individual steps.

### 27.3 Edit transparency
- In Code Mode, all changed files must appear in a visible change summary.
- Every command run must appear in the timeline.

### 27.4 Approval clarity
- Approval dialog must show action, reason, scope, and risk.
- User can deny and continue planning without task corruption.

### 27.5 Checkpoint recovery
- A checkpoint is auto-created before a multi-file edit.
- Restore operation returns files to checkpoint state.

### 27.6 Resume continuity
- Reloading the environment preserves task artifact and resume state.
- User can continue a prior task without rereading the entire transcript.

### 27.7 Context transparency
- User can inspect current context items and remove at least one optional item.
- UI shows when compaction occurred.

### 27.8 Accessibility
- All major actions are keyboard reachable.
- Screen reader labels exist for mode switcher, approvals, context list, and task steps.

## 30. UX Anti-Patterns to Avoid

The implementation SHALL avoid:
- silently editing in a read-only experience,
- hiding commands or file changes inside prose,
- forcing users to reread long transcript history to resume work,
- overusing modal popups for every low-risk action,
- collapsing all tool failures into generic “something went wrong,”
- cluttering the primary interface with every advanced control,
- decorative or childish design language inappropriate for engineering workflows,
- treating all context as invisible magic.

---

## 31. Normalized Inspirations Incorporated

This specification intentionally incorporates and reconciles the following recurring patterns observed across the source systems:

- explicit or implied **mode-based interaction** for asking, planning, coding, debugging, and reviewing,
- **slash command** and command palette style discoverability,
- **context mention** patterns for precise grounding,
- **artifact-first workflow** for plans, diffs, and checkpoints,
- **visible tool usage and approvals**,
- **resume/checkpoint continuity**,
- **instruction layering** across user/team/repo/path/task scopes,
- **custom skills/agents/commands** as reusable UX accelerators,
- **progressive disclosure** of advanced internals,
- **human-in-the-loop autonomy** with readable state and review points.

The implementation team SHALL treat these as convergent design patterns rather than provider-specific mimicry.

---

## 32. Final Recommendation

The recommended product posture is:

> Build a **chat-first engineering workspace** where conversation remains the entry point, but execution is grounded in visible artifacts, explicit modes, contextual transparency, and reversible operations.

If trade-offs are required, prioritize in this order:

1. trust and safety of actions,
2. clarity of current operating state,
3. plan/review quality,
4. low cognitive load,
5. power-user speed,
6. aesthetic polish.

A coding assistant with slightly fewer features but strong state clarity, checkpointing, plan quality, and readable tool/action UX will outperform a superficially more powerful assistant that behaves opaquely.

## 34. Appendix B – Example UX Copy Guidelines

Recommended UI copy style:
- factual,
- concise,
- operational,
- respectful of engineering context.

Examples:
- “Read-only mode: I can inspect and explain, but I will not modify files.”
- “I created a plan with 5 steps. Review it before I make changes.”
- “I need approval to run tests because this command may execute project scripts.”
- “I compacted older chat history and preserved the active plan, checkpoint, and validation summary.”
- “I could not verify the fix because the integration test requires missing environment variables.”

Avoid copy such as:
- “I’m super excited to help you code!”
- “Don’t worry, I’ve got this!”
- “Oopsie, something went wrong.”

## 35. Appendix C – Example Minimal Interaction Flow

### Scenario: fix failing test

1. User enters: `Fix the failing billing tests`.
2. Assistant starts in Plan Mode or suggests it.
3. Assistant creates plan artifact:
   - inspect latest failures,
   - identify affected files,
   - patch implementation,
   - run targeted tests,
   - summarize risk.
4. User approves.
5. Assistant creates checkpoint.
6. Assistant reads latest diagnostics and relevant files.
7. Assistant edits two files.
8. Assistant runs targeted tests.
9. Assistant renders diff summary and verification artifact.
10. Assistant offers next step: “Run full billing suite?”

This flow is the target behavior model for non-trivial engineering tasks.

# AI Coding Assistant Enhancement Specification
## Feature: Sandbox / Separated Work Environment

**Document Type:** Implementation-ready requirements specification with embedded design recommendations  
**Target Audience:** Engineering leads, backend engineers, IDE/extension engineers, platform engineers, security engineers, QA, DevOps  
**Status:** Ready for implementation planning  
**Scope:** Enhancements to an existing coding assistant  
**Normative Language:** The terms **MUST**, **SHOULD**, **MAY**, and **MUST NOT** are normative.

---

## 1. Purpose

This document defines the requirements, architecture, workflows, controls, APIs, and acceptance criteria for introducing a **Sandbox / Separated Work Environment** capability into an existing coding assistant.

The capability MUST let the assistant perform autonomous or semi-autonomous work in an environment that is **isolated from the user’s primary machine and working copy**, while still remaining practical for real development workflows.

The design is normalized from recurring patterns across major coding-assistant systems and adjacent agent platforms, including:
- OpenAI Codex: approvals, sandboxing, network controls, layered instruction files, resumable work, task artifacts
- Claude Code: hierarchical settings, permissioned tool execution, hooks, subagents, project memory
- GitHub Copilot / Copilot coding agent: agent execution in GitHub Actions-powered environments, custom instructions, hooks, runner customization, firewalls
- Aider: narrow editable scope, Docker-based separated execution option, Git-backed reversibility
- Cline: human-in-the-loop approvals, auto-approve controls, subagent isolation
- Continue: tool permission policies, agent tool loops, MCP-based extensibility
- Roo Code: auto-approval controls, mode-scoped tool permissions, checkpoints, request/cost limits

The resulting design is **provider-agnostic** and avoids provider-specific contradictions by expressing the common engineering pattern behind the products rather than mirroring one product’s UX literally.

## 2. Problem Statement

Current coding assistants often operate directly inside the user’s local working directory with broad access to:
- source files
- terminal commands
- local environment variables
- package managers
- browser/network access
- Git credentials
- external tools and MCP servers

That default model creates several risks:
- accidental destructive changes to the primary workspace
- unsafe or over-broad shell command execution
- leakage of secrets from host environment variables or credential stores
- unbounded package installation or network egress
- accidental mutation of unrelated files outside the task scope
- inability to reproduce what the assistant actually saw and executed
- weak governance for enterprise deployment

A separated work environment is intended to solve those issues while also improving:
- reproducibility
- undoability
- auditability
- policy enforcement
- team-wide consistency
- support for long-running or background agent tasks

---

## 3. Product Goals

The feature implementation MUST satisfy the following goals.

### 3.1 Primary Goals
1. Allow the assistant to execute work inside a **controlled isolated workspace** rather than directly in the user’s main environment.
2. Ensure file system, network, process, secrets, and tool access are **policy-governed**.
3. Provide **graduated trust levels** so low-risk tasks can run with less friction and high-risk tasks require explicit approval.
4. Preserve a strong **developer review loop** through diffs, logs, checkpoints, and pull-request style review.
5. Support **local IDE workflows** and **remote/hosted execution workflows** under one coherent policy model.
6. Make the environment **repeatable and reconstructable** from configuration, task metadata, and logs.

### 3.2 Secondary Goals
1. Support enterprise deployment with centralized policy controls.
2. Support team-shared configuration checked into source control where appropriate.
3. Keep the feature usable by individual developers without requiring full enterprise infrastructure.
4. Integrate cleanly with Git, GitHub, tests, CI, and MCP tools.
5. Support gradual rollout without breaking existing local workflows.

### 3.3 Non-Goals
This feature does **not** aim to:
- provide perfect isolation equivalent to a hardened hostile multi-tenant cloud sandbox by default
- replace enterprise endpoint security
- execute arbitrary unreviewed production operations
- remove the need for code review or CI validation
- bypass organization policies for package access, credentials, or repositories

## 4. Core Design Principles

The implementation MUST follow these principles.

### 4.1 Least Privilege by Default
The environment MUST start with the lowest reasonable privileges:
- read-only where possible
- no secrets by default
- restricted network by default
- minimal tool exposure by default
- explicit escalation for side-effecting actions

### 4.2 Separation of Concerns
The system MUST separate:
- the **host environment** from the **sandbox environment**
- **read access** from **write access**
- **planning** from **execution**
- **low-risk commands** from **high-risk commands**
- **user-approved policy** from **agent-discovered intentions**

### 4.3 Deterministic Policy Enforcement
Safety-critical checks MUST NOT rely only on prompt instructions. They MUST be enforced by code-level policies, wrappers, hooks, or runtime controls.

### 4.4 Reproducibility
A task SHOULD be reproducible using:
- environment specification
- mounted inputs
- policy configuration
- tool inventory
- command and file-change logs
- Git revision / task branch / PR metadata

### 4.5 Reviewability
The final product of autonomous execution MUST be easy for a human to review through:
- diffs
- command logs
- test results
- artifacts
- pull requests
- checkpoints or rollback points

### 4.6 Provider-Agnostic Execution
The sandbox system MUST be independent from the LLM vendor. The same runtime, policies, and approvals SHOULD work regardless of model provider.

## 5. User Stories

### 5.1 Local IDE User
As a developer, I want the assistant to work on my task in an isolated copy of my repo so it cannot accidentally damage my main workspace.

### 5.2 Security-Conscious User
As a developer, I want shell access, network egress, package installation, and secret usage to require explicit policy and approvals.

### 5.3 Team Lead
As a team lead, I want shared project rules for what the assistant may run, what directories it may edit, and what validations must pass.

### 5.4 Enterprise Admin
As an administrator, I want centrally enforced policies that cannot be bypassed by local user settings.

### 5.5 Remote Agent Workflow User
As a developer, I want the assistant to perform work in a remote ephemeral environment and then return a branch, patch, or PR for review.

### 5.6 QA / Auditor
As an auditor, I want complete logs of files touched, commands executed, approvals granted, policies applied, and network destinations contacted.

## 6. Definitions

### 6.1 Host Environment
The user’s real local machine, shell, environment variables, credentials, editor session, and primary working copy.

### 6.2 Sandbox
An execution environment separated from the host by file system, process, network, credential, and policy boundaries.

### 6.3 Workspace Clone
A working copy of the repository created specifically for the sandbox task. This MAY be a Git worktree, shallow clone, full clone, copy-on-write mount, or mounted branch workspace.

### 6.4 Isolation Profile
A named configuration that defines file, network, command, secret, and tool access. Examples:
- `read_only_local`
- `workspace_write_no_network`
- `workspace_write_limited_network`
- `remote_ephemeral_ci_runner`

### 6.5 Approval
A user or policy-authorized decision allowing a specific action class, exact action, or policy scope.

### 6.6 Checkpoint
A restorable sandbox state capture, usually represented by Git commit, patch, snapshot metadata, or filesystem snapshot.

### 6.7 Side-Effecting Action
Any operation that mutates files, launches processes, changes Git state, installs software, writes external systems, or makes network requests.

6. execution loop
7. validation and test execution
8. artifact packaging
9. promotion back to user via diff / patch / branch / PR

The system MUST support:
- per-tool approvals
- per-command approvals
- per-network-destination approvals
- per-directory edit permissions
- session-scoped approvals
- project-scoped approvals
- organization-managed policies
- automatic approvals for explicitly allowlisted actions

### 7.4 Artifacts
The system MUST persist and expose:
- patch/diff
- changed file list
- command log
- validation/test log
- environment manifest
- approvals granted and denied
- network access attempts

- final task summary

## 9. Sandbox Execution Model

### 9.1 Isolation Modes
The product MUST define explicit isolation modes rather than a single vague “sandbox” concept.

Recommended baseline modes:

| Mode | Description | Typical Use | Default Risk |
|---|---|---|---|
| `host_readonly` | Agent can inspect host files with no writes, no shell side effects | analysis, planning, triage | low |
| `host_workspace_write` | Agent can edit current workspace under approvals | small local edits | medium |
| `local_worktree_sandbox` | Agent works in separate Git worktree/clone | normal autonomous coding | medium-low |
| `container_workspace_sandbox` | Agent works in isolated container with mounted repo clone | package install, test runs | medium |
| `remote_ephemeral_runner` | Agent runs in isolated remote runner and returns PR/patch | long-running or organization workflows | medium-high |
| `high_restriction_review` | Read-only + no network + no installs | secure review, audit, code explanation | lowest |

The UI MUST make the active mode visible at all times.

### 9.2 Workspace Materialization
For any write-capable isolated mode, the system MUST create a dedicated task workspace using one of the following:
- Git worktree from current commit/branch
- clone into task directory
- checkout task branch in isolated directory
- remote runner clone

Recommended default for local workflows:
- use **Git worktree** when repository is already under Git and local filesystem allows it
- fall back to cloned workspace when worktree is not viable

#### Requirements
- The sandbox workspace MUST record the source repository path and source commit SHA.
- The sandbox workspace MUST use a task identifier.
- The sandbox workspace MUST NOT write into the user’s primary workspace except through explicit promotion/merge/apply operations.

Example directory layout:

```text
~/.assistant/sandboxes/
  task-2026-03-08-001/
    manifest.json
    workspace/
    logs/
    artifacts/
    checkpoints/
```

### 9.3 Filesystem Access Rules
The system MUST classify file access into:
- read-only mounts
- writable task workspace
- prohibited paths

#### Minimum Requirements
- The task workspace MUST be writable.
- The user’s home directory MUST NOT be broadly writable from the sandbox.
- Sensitive paths MUST be blocked by default, including examples such as:
  - `~/.ssh`
  - `~/.aws`
  - `~/.config/gcloud`
  - `~/.npmrc`
  - OS credential stores
  - unrelated repositories
  - system directories
- The sandbox MUST support an allowlist of extra readable paths if the user explicitly grants them.

### 9.4 Process Isolation
The sandbox SHOULD isolate process execution from host processes where practical.

Minimum requirements:
- spawned commands MUST run through an execution wrapper
- command metadata MUST be logged
- allowed/disallowed decisions MUST be enforced before process launch
- child process trees MUST be tracked for cleanup
- runaway processes MUST be killable by timeout or cancellation

### 9.5 Network Isolation
Network access MUST be policy-governed.

Supported states MUST include:
- `deny_all`
- `allow_none_but_localhost`
- `allowlist_domains`
- `allowlist_hosts_and_ports`
- `full_access_with_warning`

Examples of allowed destinations by policy profile:
- package registries
- documentation sites
- internal artifact mirrors
- Git hosting
- issue trackers
- approved MCP endpoints

The system MUST log every attempted outbound network destination when technically feasible.

### 9.6 Secret Handling
Secrets MUST be treated as separate governed resources.

Requirements:
- host environment variables MUST NOT be inherited wholesale
- only explicitly mapped secrets MAY be exposed
- secrets SHOULD be mounted as ephemeral environment variables or files with short lifetime
- secrets MUST be redacted from logs
- prompts MUST NOT include secrets unless the user explicitly requests it and policy permits it
- secret exposure events MUST be auditable

### 9.7 Tool Isolation
All tools exposed to the model MUST be declared and mediated.

Tool categories:
- file read/search tools
- file write/edit tools
- shell/terminal tool
- git tool
- test/build tool
- browser/web-fetch tool
- network access tool
- MCP tool bridge
- issue/PR tools

The model MUST NOT have any hidden side-effecting capability outside the declared tool contract.

## 10. Approval Model

### 10.1 Approval Tiers
The implementation MUST support the following approval tiers.

#### Tier A: Implicitly Safe
Examples:
- read file
- list directory
- search code
- inspect git status
- inspect test logs

These MAY be auto-approved by default.

#### Tier B: Controlled Write
Examples:
- edit files in writable sandbox workspace
- create new files in workspace
- run formatter inside sandbox
- execute read-mostly tests without network

These SHOULD be allowed automatically only inside explicit write-capable sandbox modes.

#### Tier C: Elevated Local Side Effects
Examples:
- install packages
- run migrations against local services
- modify git branches/tags
- execute arbitrary shell commands
- call MCP tools with external side effects

These MUST require explicit policy and SHOULD require user approval unless allowlisted by trusted policy.

#### Tier D: External or Irreversible Operations
Examples:
- push branch to remote
- open or update PR
- publish package
- deploy infrastructure
- write to ticketing systems
- delete files recursively
- alter system configuration

These MUST require explicit approval every time unless an organization-managed policy says otherwise and the action is constrained to a reviewed workflow.

### 10.2 Approval Granularity
The user MUST be able to approve actions at multiple granularities:
- once
- for current session
- for this task only
- for this repository/project
- for this exact command pattern
- never allow

### 10.3 Approval Object Model

```ts
export type ApprovalScope =
  | "once"
  | "task"
  | "session"
  | "project"
  | "org-policy";

export type ActionKind =
  | "read_file"
  | "write_file"
  | "exec_command"
  | "network_request"
  | "install_package"
  | "git_push"
  | "open_pr"
  | "mcp_call";

export interface ApprovalDecision {
  actionKind: ActionKind;
  matcher: string;           // exact command, glob, regex, host, path, tool name, etc.
  scope: ApprovalScope;
  allow: boolean;
  createdAt: string;
  createdBy: "user" | "admin" | "policy";
  expiresAt?: string;
  reason?: string;
}
```

### 10.4 Policy over Prompt
Approval decisions MUST be enforced by the runtime and MUST NOT depend on the model promising to behave.

## 11. Policy Engine Requirements

The system MUST provide a policy engine with deterministic evaluation.

### 11.1 Policy Domains
The engine MUST support at least:
- path access policy
- command execution policy
- network egress policy
- secret exposure policy
- tool usage policy
- Git operation policy
- artifact retention policy

### 11.2 Policy Sources
Policies MUST be resolved from layered sources, from highest to lowest precedence:
1. organization-managed policy
2. repository/project shared policy
3. user local policy
4. task-specific policy
5. interactive approval decisions

Interactive approval MUST NOT override organization-denied actions.

### 11.3 Path Policy
Example:

```yaml
paths:
  writable:
    - workspace/**
    - artifacts/**
  readable:
    - workspace/**
    - docs/**
    - .github/**
  denied:
    - ~/.ssh/**
    - ~/.aws/**
    - ~/.config/gcloud/**
    - /etc/**
    - ../**
```

### 11.4 Command Policy
The command policy MUST classify commands into:
- read-only commands
- build/test commands
- dependency management commands
- VCS commands
- destructive commands
- networked commands

Example:

```yaml
commands:
  allow:
    - "git status"
    - "git diff *"
    - "npm test *"
    - "pytest *"
    - "ruff check *"
    - "eslint *"
  ask:
    - "npm install *"
    - "pnpm install *"
    - "docker build *"
    - "git push *"
  deny:
    - "rm -rf / *"
    - "sudo *"
    - "chmod -R 777 *"
    - "curl * | bash"
    - "powershell -EncodedCommand *"
```

### 11.5 Network Policy

```yaml
network:
  mode: allowlist
  allowedHosts:
    - registry.npmjs.org
    - pypi.org
    - files.pythonhosted.org
    - api.github.com
    - github.com
  deniedHosts:
    - metadata.google.internal
    - 169.254.169.254
```

### 11.6 Policy Evaluation API

```ts
export interface PolicyContext {
  taskId: string;
  repoId: string;
  isolationMode: string;
  userId: string;
  command?: string;
  path?: string;
  host?: string;
  toolName?: string;
  actionKind: ActionKind;
}

export interface PolicyResult {
  decision: "allow" | "ask" | "deny";
  matchedRule?: string;
  reason: string;
  requiresApproval: boolean;
}

export interface PolicyEngine {
  evaluate(ctx: PolicyContext): Promise<PolicyResult>;
}
```

## 13. Command Execution Wrapper

All process execution MUST go through a wrapper layer.

### 13.1 Responsibilities
The wrapper MUST:
- normalize command representation
- evaluate policy before execution
- request approval when needed
- set timeouts
- capture stdout/stderr
- mask secrets in output
- track duration and exit code
- log command provenance

### 13.2 Execution Contract

```ts
export interface ExecRequest {
  sandboxId: string;
  command: string;
  cwd?: string;
  env?: Record<string, string>;
  timeoutMs?: number;
  reason?: string;
}

export interface ExecResult {
  command: string;
  exitCode: number;
  stdout: string;
  stderr: string;
  durationMs: number;
  approvalMode: "auto" | "interactive" | "policy";
  policyRule?: string;
}
```

### 13.3 High-Risk Command Detection
The system MUST classify high-risk commands before execution.

Examples of high-risk categories:
- recursive delete
- privilege escalation
- shell pipelines downloading and executing code
- branch force-push
- destructive database operations
- secret-printing patterns
- broad chmod/chown changes
- uncontrolled package install scripts

Example classification helper:

```ts
const HIGH_RISK_PATTERNS: RegExp[] = [
  /\brm\s+-rf\b/i,
  /\bsudo\b/i,
  /curl\b.*\|\s*(bash|sh|zsh|powershell)/i,
  /wget\b.*\|\s*(bash|sh|zsh|powershell)/i,
  /git\s+push\s+.*--force/i,
  /DROP\s+DATABASE/i,
  /TRUNCATE\s+TABLE/i,
];

export function isHighRiskCommand(cmd: string): boolean {
  return HIGH_RISK_PATTERNS.some((re) => re.test(cmd));
}
```

### 13.4 Recommended UX
Before a risky action, the UI SHOULD show:
- exact command
- working directory
- why the agent wants to run it
- affected paths or destination hosts
- policy basis
- approval options

## 14. Secrets and Credentials Model

### 14.1 Secret Sources
The system MAY support secrets from:
- local secure store
- environment variable mapping
- OS keychain
- cloud secret manager
- CI secret store

### 14.2 Secret Injection Rules
- Secrets MUST be explicitly named.
- Secrets MUST only be injected into tasks that require them.
- Secrets SHOULD be scoped to the smallest feasible environment.
- Secrets MUST be removable at task end.
- Secrets MUST NOT be visible to the model as plain text unless unavoidable for a declared operation.

### 14.3 Secret Redaction
All logs and assistant-visible tool outputs MUST be redacted using:
- exact secret match redaction
- high-entropy token heuristics where practical
- provider token pattern matching where available

Example:

```ts
export function redactSecrets(text: string, secrets: string[]): string {
  return secrets.reduce((acc, secret) => {
    if (!secret) return acc;
    return acc.split(secret).join("[REDACTED]");
  }, text);
}
```

### 14.4 Credential Containment
The sandbox MUST NOT inherit the user’s full git credential helper state by default.

Recommended pattern:
- use a scoped token for remote operations
- inject it only for remote PR/push tasks
- prefer short-lived credentials

## 15. Network and External Resource Control

### 15.1 Default
The default write-capable sandbox SHOULD start with `allowlist` network policy or stricter.

### 15.2 External Access Classes
The system MUST distinguish between:
- package registry access
- documentation browsing
- Git hosting
- issue/PR APIs
- artifact download
- MCP endpoints
- arbitrary browsing

### 15.3 Firewall / Proxy Integration
For enterprise environments, the product SHOULD support:
- forced HTTP/S proxy
- domain allowlists
- DNS filtering
- egress logs
- custom CA or SSL interception compatibility where required by enterprise policy

### 15.4 Metadata Endpoint Protection
Cloud metadata endpoints and equivalent privileged internal endpoints MUST be denied by default.

## 16. Git, Branching, and Promotion Back to User

The sandbox feature MUST integrate tightly with Git.

### 16.1 Workspace Branch Strategy
For any write-capable task, the system SHOULD create:
- task branch in sandbox workspace
- checkpoint commits or patches during execution

Example naming:
- `assistant/task/2026-03-08-001`
- `assistant/fix/login-timeout-001`

### 16.2 Promotion Models
The system MUST support one or more of:
- apply patch to host workspace
- merge sandbox worktree back into host branch
- push sandbox branch to remote
- open pull request
- export patch bundle

### 16.3 Review Gate
The final step SHOULD be review-first by default:
- show diff summary
- run validations
- display test outcomes
- require user confirmation before host merge or remote push

### 16.4 Checkpoints
The product MUST support checkpoint creation before major stages.

Checkpoint triggers SHOULD include:
- before package install
- before formatter bulk changes
- before refactor across many files
- before migration generation
- before remote push

Example:

```ts
async function createGitCheckpoint(workspacePath: string, label: string): Promise<void> {
  await execa("git", ["add", "-A"], { cwd: workspacePath });
  await execa("git", ["commit", "-m", `[checkpoint] ${label}`], { cwd: workspacePath });
}
```

## 17. Hooks, Guardrails, and Deterministic Validation

The system SHOULD support deterministic lifecycle hooks, similar in spirit to best-in-class coding agent systems.

### 17.1 Hook Points
Recommended hook events:
- `onTaskStart`
- `beforeCommand`
- `afterCommand`
- `beforeWrite`
- `afterWrite`
- `beforePromotion`
- `beforeDestroy`
- `onPolicyDecision`
- `onApprovalRequested`

### 17.2 Use Cases
Hooks MAY be used for:
- enforce formatting
- run linting
- deny prohibited command patterns
- validate that touched files are within allowed scope
- inject standard environment preparation
- notify users
- persist custom audit data

### 17.3 Hook Contract

```ts
export interface HookContext {
  taskId: string;
  sandboxId: string;
  event: string;
  workspacePath: string;
  payload: Record<string, unknown>;
}

export interface HookResult {
  decision?: "allow" | "deny";
  message?: string;
  annotations?: Record<string, string>;
}
```

### 17.4 Deterministic over Advisory
If a project requires `npm test` or `pytest` before promotion, that MUST be a hook/policy rule, not just a prompt suggestion.

## 18. Subtasks and Subagents

If the assistant supports subagents or delegated subtasks, each subagent SHOULD run in its own isolated context window and SHOULD NOT automatically inherit all capabilities.

### 18.1 Subagent Rules
- read-only subagents SHOULD be the default
- subagents MUST NOT inherit write permissions unless explicitly required
- subagents SHOULD NOT get network access unless explicitly needed
- subagents SHOULD return compact findings rather than modifying the parent’s workspace directly

### 18.2 Rationale
This pattern reduces blast radius and preserves the main sandbox for actual implementation steps.

## 19. MCP / External Tool Integration

### 19.1 Tool Mediation
External MCP or plugin tools MUST be mediated through the same policy and approval system as built-in tools.

### 19.2 Required Controls
For each external tool, the system MUST define:
- tool identity
- side-effect classification
- allowed input scopes
- approval tier
- secret access requirements
- network implications

### 19.3 Resource vs Action Distinction
The policy model SHOULD distinguish:
- resource fetch/read operations
- action or mutation operations

### 19.4 Example
```ts
export interface ToolCapabilityDescriptor {
  name: string;
```

## 24. Security Requirements

### 24.1 MUST Requirements
1. The model MUST NOT receive unrestricted shell access by default.
2. The sandbox MUST NOT inherit all host environment variables by default.
3. The sandbox MUST NOT have broad writable host filesystem access.
4. Destructive commands MUST be denylisted or require explicit approval.
5. Metadata endpoints and privileged local endpoints MUST be blocked by default.
6. Promotion from sandbox back to host or remote MUST be reviewable.
7. Organization-managed deny rules MUST be non-bypassable by normal users.
8. Secret values MUST be redacted from logs.

### 24.2 SHOULD Requirements
1. Prefer ephemeral environments for remote autonomous tasks.
2. Prefer read-only subagents.
3. Prefer short-lived credentials for remote operations.
4. Prefer allowlist network policy over unrestricted internet access.
5. Prefer deterministic hooks for mandatory validations.

## 27. Example TypeScript Skeleton

```ts
export class SandboxOrchestrator {
  constructor(
    private readonly sandboxManager: SandboxManager,
    private readonly policyEngine: PolicyEngine,
    private readonly approvals: ApprovalService,
    private readonly exec: ExecService,
    private readonly audit: AuditService,
  ) {}

  async runTask(task: TaskRequest): Promise<TaskResult> {
    const sandbox = await this.sandboxManager.create({
      taskId: task.id,
      repoPathOrRemote: task.repoPath,
      baseRef: task.baseRef,
      isolationMode: task.isolationMode,
      policyBundleId: task.policyBundleId,
      writable: task.writable,
      networkMode: task.networkMode,
      allowedHosts: task.allowedHosts,
      injectedSecrets: task.secretIds,
    });

    await this.audit.log({
      timestamp: new Date().toISOString(),
      taskId: task.id,
      sandboxId: sandbox.sandboxId,
      eventType: "sandbox.created",
      actor: "system",
      details: { mode: task.isolationMode },
    });

    try {
      return await this.executePlan(task, sandbox);
    } finally {
      if (task.cleanup !== "retain") {
        await this.sandboxManager.destroy(sandbox.sandboxId);
      }
    }
  }

  private async guardedCommand(
    sandboxId: string,
    task: TaskRequest,
    command: string,
    reason: string,
  ): Promise<ExecResult> {
    const policy = await this.policyEngine.evaluate({
      taskId: task.id,
      repoId: task.repoId,
      isolationMode: task.isolationMode,
      userId: task.userId,
      command,
      actionKind: "exec_command",
    });

    if (policy.decision === "deny") {
      throw new Error(`Blocked by policy: ${policy.reason}`);
    }

    if (policy.decision === "ask") {
      const approved = await this.approvals.request({
        taskId: task.id,
        kind: "exec_command",
        matcher: command,
        reason,
      });
      if (!approved) {
        throw new Error("User denied command approval");
      }
    }

    return this.exec.run({
      sandboxId,
      command,
      reason,
      timeoutMs: 300000,
    });
  }
}
```

## 31. Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Sandbox too restrictive for real work | poor adoption | allow named profiles and controlled escalation |
| Users bypass sandbox for convenience | security regression | make sandbox fast and default for write tasks |
| Logs expose secrets | severe | aggressive redaction + secret isolation |
| Policy complexity confuses users | usability issues | provide presets, clear UI, and audit messages |
| Container startup latency is high | slow UX | warm images, worktree-first local mode |
| Remote runner cost grows | budget issues | task/runtime/request limits |
| Agent still damages task workspace | medium | checkpoints, Git reversibility, review-first promotion |
| Hidden side effects through external tools | severe | tool capability descriptors + policy mediation |

## 32. Design Decisions Recommended for the Implementation Team

The following decisions are strongly recommended.

1. **Default all autonomous write tasks to a separate worktree sandbox.**  
   This gives major safety gains with moderate implementation cost.

2. **Treat shell access as a policy-enforced tool, not a direct model privilege.**  
   This is non-negotiable for safe operation.

3. **Separate read-only analysis from write-capable execution.**  
   Many tasks do not need write access and should stay in low-risk mode.

4. **Make network access explicit and visible.**  
   A silent internet-connected coding agent is difficult to govern.

5. **Require review before promotion from sandbox to host/remote.**  
   The sandbox reduces damage risk; the review gate preserves developer agency.

6. **Use deterministic hooks for mandatory validations.**  
   Do not rely on the model to remember to test or lint.

7. **Adopt a layered configuration model.**  
   Team rules and organization rules need durable precedence.

8. **Represent all side-effecting capabilities as typed, auditable actions.**  
   This is essential for observability and policy enforcement.

9. **Use ephemeral remote runners for long-running autonomous tasks.**  
   This minimizes workstation impact and improves reproducibility.

10. **Keep the design provider-agnostic.**  
    The runtime, policies, and logs should survive model/provider changes.

## 37. Appendix D: Final Summary

The recommended implementation is a **graduated sandbox architecture** centered on a default **separate worktree sandbox**, deterministic **policy enforcement**, explicit **approval controls**, auditable **command and network mediation**, and review-first **promotion back to Git/GitHub workflows**.

This approach captures the strongest recurring best practices seen across modern coding assistants without copying any one product too literally:
- isolated execution rather than unconstrained host mutation
- explicit tool permissions rather than hidden capabilities
- reviewable diffs rather than opaque autonomy
- project and organization policy layers rather than ad hoc prompts
- reproducible environment manifests rather than implicit local state

That combination is the correct foundation for a serious coding assistant intended for real-world engineering teams.

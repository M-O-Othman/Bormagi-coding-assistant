# Implementation Plan: Agent Knowledge Base Design v1.0 (Revised)

This plan is revised based on user feedback to prioritize simplicity, local VS Code extension constraints (no Docker, no Redis, no Ollama), and to skip the tool manifest migration.

## Phase 1: L2 Knowledge Base & RAG Enforcements
Objective: Utilize simple file-based and local-vector storage. Ensure agents definitively consult the knowledge base (folders or embeddings) before answering user queries.

1. **Embedding & Vector Services (Local Constraints)**
   - Lean on the existing `LocalEmbeddingProvider` (or Provider-based embeddings using the user's existing API keys) and the local JSON-based `VectorStore.ts`. No ChromaDB or Ollama will be introduced.
   - Refine the `KnowledgeManager` and `RetrievalService` to handle user choices for storing/retrieving knowledge (direct folder search vs vector embeddings).
2. **Agent Query Enforcement**
   - Update `AgentRunner.ts` to strictly enforce that the agent retrieves and considers context from the knowledge base *before* generating its response. Ensure the system prompt dynamically forces adherence to retrieved documentation.
   - Inject the retrieved context into the LLM context envelope reliably.

## Phase 2: L3 Semantic Memory & Decision Log
Objective: Build out the missing episodic memory and decision logs.

1. **Decision Record Management**
   - Build `DecisionManager.ts` to manage creation, resolution, and updating of `DecisionRecord` objects stored locally in the agent's memory directories.
2. **Memory Consolidation Process**
   - Implement `Consolidator` that runs after sessions to merge `TurnMemory` into structured `EpisodicMemory` (action items, main points).
   - Extract `SemanticFacts` automatically, track confidence, and evict outdated facts using the existing `PromotionEngine`.

## Phase 3: L4 Multi-Agent Collaboration
Objective: Enable agents to delegate tasks and communicate asynchronously using simple local files.

1. **Message Bus Implementation**
   - Create `FileMessageBus` implementing `MessageBusClient` in `src/collaboration/bus.ts` using simple JSON files and `chokidar` for watching the local `shared/bus/{agentId}/inbox` directories. No Redis required.
2. **Agent Registry Enhancement**
   - Update `AgentRegistry.ts` to enforce the `can_delegate_to` and `can_receive_from` attributes.
3. **Delegation Protocol**
   - Create a `DelegationManager` (or integrate into `AgentRunner`) to format `task_delegation` messages, handle agent acceptance/rejection, and route results back. The agent will prepare responses based on RAG knowledge to fulfill delegation requests.
4. **Shared Knowledge Push/Pull**
   - Empower agents to broadcast decisions to all other listening agents via the `knowledge_share` message type.

## Phase 4: Phase Skipped
*Tool Manifest Isolation and Sandboxing (L5) is skipped for now based on user feedback.*

## Verification & Testing

- Run an integration test spinning up two agents communicating via the message bus, delegating a simulated task and consulting the file-based knowledge store.

# Open Questions
**Feature:** AI Coding Assistant Sandbox Separated Work Environment

Before executing the implementation plan, the following details require clarification to ensure we match the product/design expectations:

1. **Git Assumption for Local Sandbox:** For the `local_worktree_sandbox` mode, should we assume the user's workspace is ALWAYS a Git repository (and use `git worktree`), or must we build a graceful fallback that just copies the directory structure if no Git repository is found?
**Answer** Build a fallback
2. **Review UX for Promotion:** For Phase 1, does the promotion review (showing the diffs before applying to the host) need to be a brand new bespoke UI view inside the Bormagi dashboard, or is it acceptable to leverage standard VS Code diff editors and a standard Yes/No prompt to start?
**Answer** Use vscode diff editor and yes/no prompt but allow accept all option as well in relevant conditions

3. **Policy Engine Configuration:** The spec recommends layered configuration (Org -> Project -> Local User). For a local VS Code user, should the default `.bormagi/policies/sandbox.policy.yaml` be auto-generated upon workspace initialization if it doesn't exist?
**Answer** ignore this for now , use one layer (project)
4. **Execution Wrapper vs MCP Servers:** Currently, shell execution and file operations happen inside separate Built-in MCP Servers (e.g., `terminal-server.ts`, `filesystem-server.ts`). Should the `PolicyEngine` and `guardedCommand` logic be injected directly into these MCP servers, or should the `AgentRunner` intercept and mediate the tool calls *before* they even reach the MCP Host? (Intercepting at the `AgentRunner` level securely shields all external MCP tools as well).
**Answer**use your judgement and implement the preferable option , remember "if it is too complicated it is propably not the right option"

# Implementation Plan
## Feature: AI Coding Assistant Chatbot AI / UX
### Spec: `7.ai_coding_assistant_chatbot_ai_ux_spec.md`
### Date: 2026-03-08

---

## Guiding Principles

- Preserve all existing working functionality (streaming, agent switching, token stats, approval cards, checkpoints, workflow commands).
- Each phase is independently deployable and leaves the extension in a consistent, working state.
- Changes are additive — new message types and UI components extend the existing `MessageToWebview` union and `chat.html` rather than replacing them.
- Backend changes are minimal: the backend services are solid; the primary work is wiring them to the front end.
- All UI additions must use VS Code CSS custom properties (no hardcoded colours).

---

## Pre-work: Mode Naming Alignment

**Issue:** The spec requires modes `ask | plan | code | debug | review` (§6.1). The current implementation uses `explain | edit | debug | review | plan | search | test-fix` (in `src/context/types.ts` and `ModeClassifier.ts`).

**Decision needed (see open questions):** Whether to rename the existing backend modes or keep them and map them to spec names only in the UI layer. The plan below assumes a **UI alias layer** — no breaking rename of backend types in Phase 1, allowing gradual migration.

## Phase 1 — Foundational UX (highest priority)

Deliverables: mode switcher in composer, structured approval prompts, action timeline cards, basic context rail, diff summary in chat, compaction notice, empty-state examples.

### 1.1 Mode Switcher in Composer

**Files to change:**
- `media/chat.html` — add mode selector control to composer footer
- `src/chat/ChatController.ts` — add `mode_changed` message type
- `src/chat/ChatViewProvider.ts` — pass current mode from webview to controller

**Implementation:**
1. Add a `MessageToWebview` entry: `{ type: 'mode_changed'; mode: string }`.
2. Add a `MessageFromWebview` entry (handled in `ChatViewProvider`): `{ type: 'set_mode'; mode: string }`.
3. In `chat.html` composer footer, add a `<select id="mode-sel">` with options for `ask | plan | code | debug | review`. Initially map:
   - `ask` → internal `explain`
   - `code` → internal `edit`
   - others pass through directly.
4. Show a read-only badge ("Read-only") in the composer when mode is `ask`.
5. When user changes mode in `ask` and types an edit-intent message, show an inline suggestion to switch to `code` mode.
6. Persist selected mode in webview state (survives page refresh via `vscode.getState()`).

**Test:** `src/tests/unit/ChatController.test.ts` — verify `mode_changed` message emitted; verify `set_mode` forwarded to `ModeClassifier.buildUserModeDecision`.

### 1.2 Structured Approval Prompts (§10.2)

**Files to change:**
- `src/chat/ChatController.ts` — enrich `action_request` message
- `media/chat.html` — update action card rendering

**Implementation:**
1. Extend `action_request` in `MessageToWebview`:
   ```ts
   | { type: 'action_request'; id: string; prompt: string; actions: string[];
       kind?: 'edit' | 'command' | 'network' | 'git' | 'external_tool';
       reason?: string; scope?: string[]; risk?: 'low' | 'medium' | 'high'; alternatives?: string[] }
   ```
2. Update `requestInlineAction()` in `ChatController` to accept optional `ApprovalMeta` parameter.
3. Update callers in `AgentRunner` to pass metadata when invoking the approval callback (risk level, scope).
4. Update `chat.html` action card HTML to render `kind`, `reason`, `scope`, `risk` fields when present.
5. Style risk levels: green for `low`, amber for `medium`, red for `high`.

**Backward compatibility:** All existing `action_request` messages without optional fields continue to render identically.

**Test:** `src/tests/unit/ChatController.test.ts` — verify structured fields forwarded; `src/tests/unit/ApprovalCard.test.ts` (new) — verify rendering with and without optional fields.

4. Plan cards update in place when `updateMilestone()` is called (new `plan_milestone_updated` message).

**Test:** `src/tests/integration/plan-artifact-rendering.test.ts` (new) — verify plan card rendered from `ExecutionPlan`; verify milestone update updates card state.

## Integration Impacts

1. **`AgentRunner.run()` signature**: New callbacks added (`onCompaction`, `onPlanCreated`, `onDiffSummary`, `onCheckpointCreated`, `onContextUpdate`). All new callbacks are optional to preserve backward compatibility with direct callers in tests.

2. **`ChatController.handleUserMessage()`**: New slash commands routed before the generic message path. All new branches are additive.

3. **`MessageToWebview` union**: Extended with new discriminated union members. Existing `switch` statements in `ChatViewProvider` and `MainPanel` that handle messages need `default` branches — verify these exist (they do in `ChatViewProvider`).

4. **`chat.html`**: New HTML sections added (context rail, plan card, diff card, mode selector). The single-file approach is maintained; new components follow the existing inline-HTML-building pattern.

5. **No changes to**: `ModeClassifier` mode values (backward compat), `PlanManager` disk format, `CheckpointManager` storage, `PolicyEngine` policy definitions.

---

## Backward-Compatibility Considerations

- All new fields in `action_request` are optional — existing `action_request` messages without them render unchanged.
- New slash commands do not conflict with existing `/undo`, `/artifact`, `/wf-*` commands.
- New `MessageToWebview` types are ignored by any subscriber that does not handle them (`ChatViewProvider` posts to webview; `MainPanel` selectively forwards).
- The mode alias layer (`ask` ↔ `explain`, `code` ↔ `edit`) is UI-only and does not change backend mode logic.

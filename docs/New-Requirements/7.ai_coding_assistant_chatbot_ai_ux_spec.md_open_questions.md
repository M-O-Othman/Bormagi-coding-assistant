## OQ-1 — Mode Naming Alignment (Blocking)

**Question:** The spec defines modes as `ask | plan | code | debug | review` (§6.1). The existing backend implementation uses `explain | edit | debug | review | plan | search | test-fix` (in `src/context/types.ts`, `ModeClassifier.ts`, and prompt assembler). These names diverge on two core modes (`ask` vs `explain`, `code` vs `edit`).

Which approach should be taken?

- **Option A:** Rename the backend types to match the spec exactly (`explain` → `ask`, `edit` → `code`). This is a breaking change across `AgentRunner`, `PlanManager`, `PromptAssembler`, `ModeBudgets`, all tests, and stored plan files.
- **Option B:** Keep backend names as-is and add a UI alias layer only — the composer shows "Ask" but internally sends `explain`; shows "Code" but internally sends `edit`. No backend changes.
- **Option C:** Keep backend names and add `ask` and `code` as new valid mode values that co-exist with `explain` and `edit` (gradual migration path).

**Answer:**
>Option C

## OQ-2 — Mode Switcher Placement (Blocking for §6.3, §7.1)

However, the current `StatusBar.ts` shows only the agent name — there is no visible mode dropdown in either `chat.html` or the VS Code status bar.

Where should the mode switcher live?

- **Option B:** VS Code status bar item (existing `StatusBar.ts`) — requires `vscode.StatusBarItem` commands, less discoverable.
- **Option C:** Both: small pill in composer + full switcher in status bar.

**Answer:**
> option C

## OQ-3 — Ask Mode Enforcement Scope (Blocking for §6.1.1, §27.1)

**Question:** The spec requires Ask Mode to be "disallowed by default" for file edits and command execution. The current `PolicyEngine` and `AgentRunner` do not distinguish ask-mode restrictions from general approval-required operations.

How strictly should Ask Mode block edits?

- **Option A:** Fully enforce at the `AgentRunner` level — if mode is `ask`/`explain`, the `ExecWrapper` and file-write tools return a hard error. Requires policy engine extension.
- **Option B:** Enforce only in the composer UI — the send button shows a warning if the user appears to be requesting an edit while in Ask Mode, but no backend block.
- **Option C:** Enforce at both layers — UI warning + backend soft block (assistant self-enforces via system prompt instruction).

**Answer:**
> option B

**Question:** Full WCAG compliance for a VS Code webview is constrained by the VS Code platform itself (some focus management is handled by VS Code's host). The spec requires "modern accessibility standards."

- **Option B:** Full WCAG 2.1 AA compliance including color contrast audit, reduced-motion support, and screen-reader testing.
- **Option C:** Defer accessibility to Phase 4 as planned; Phase 1 adds no accessibility work.

**Answer:**
option C

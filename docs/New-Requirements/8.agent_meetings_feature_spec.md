
# Agent Meetings Feature Specification
**Product:** Local VS Code extension-based AI coding assistant  
**Feature:** Agent Meetings  
**Primary stack:** TypeScript + Node.js  
**Execution model:** Local extension host + local webview UI  
**LLM model routing:** Configurable per agent/provider  
**Knowledge model:** Per-agent folder-based knowledge, RAG retrieval, and embedded vector datastore  
**Document type:** Implementation-ready requirements and design specification  
**Status:** Draft for implementation  
**Authoring intent:** This document is written to be handed directly to the implementation team.

---

## 1. Executive Summary

The Agent Meetings feature enables a set of specialized AI agents to discuss a scoped problem, produce structured recommendations, surface risks and unknowns, and present decision prompts to the human user without collapsing into unproductive “talkshop” behavior.

This feature is **not** a workflow engine. It is a bounded, human-governed, multi-agent deliberation system. It can **propose** actions and create/resolve meeting artifacts, but the separate workflow/action-items subsystem remains the system of record for execution.

This specification consolidates:
- the design intent and failure analyses contained in the uploaded meeting enhancement documents,
- the existing Bormagi structured outcome and workflow API design,
- and external best practices from official multi-agent orchestration guidance.

The design normalizes these sources into one coherent implementation approach:
- use the **lowest coordination complexity that reliably works**,
- keep agent roles sharply specialized,
- keep communication **structured, typed, and validated**,
- use **agenda-scoped context** to prevent drift,
- enforce **human decision checkpoints**,
- maintain **durable meeting state** and structured summaries,
- and treat inter-agent communication as a governed protocol rather than open-ended free chat.

---

## 2. Goals

### 2.1 Primary goals
1. Make meetings productive, bounded, and decision-oriented.
2. Prevent conversational drift, repeated restatement, and pseudo-implementation inside meetings.
3. Keep the human as the explicit decision maker.
4. Preserve strong specialization among agents.
5. Support per-agent configurable models and knowledge sources.
6. Work reliably in a local VS Code extension with recoverable state.
7. Produce machine-usable artifacts:
   - structured meeting rounds,
   - agenda summaries,
   - open questions,
   - decision prompts,
   - action proposals,
   - optional action-item emissions to the separate workflow/action system.

### 2.2 Secondary goals
1. Provide transparent UI state and explainable meeting flow.
2. Make failures diagnosable through schema validation, event logs, and replayable state.
3. Minimize token waste through scoped context assembly and structured summaries.

---

## 3. Non-Goals

The following are explicitly out of scope for this feature:

1. Full workflow orchestration across implementation stages.
2. Automated execution of development tasks as part of meetings.
3. Allowing agents to finalize product or technical decisions without the human.
4. Broad autonomous agent-to-agent delegation chains.
5. Treating meetings as a substitute for the separate action-item/workflow subsystem.

**Important boundary:**  
Meetings may create **action proposals** and may create **action items** only when allowed by policy and/or after human approval. The execution lifecycle of those action items remains outside the meeting subsystem.

---

## 4. External Design Principles Incorporated

This design is shaped by the following recurring external best practices:

1. **Start with the lowest necessary coordination complexity.**  
   Multi-agent orchestration adds latency, cost, and failure modes. Use multi-agent meetings only for cross-functional problems that truly benefit from specialization.

2. **Use clear specialization and avoid redundant agents.**  
   Agents should exist only where they add meaningful expertise, not as decorative participants.

3. **Prefer structured handoffs and structured interaction over free-form group chat.**  
   Routines, handoffs, supervisor patterns, and typed tool-mediated transitions are more controllable than uncontrolled conversation.

4. **Define explicit human-in-the-loop checkpoints.**  
   Approval, clarification, escalation, and decision points must be explicit and represented in state.

5. **Use immutable summaries and bounded state.**  
   Long-running multi-agent conversations drift unless state is compacted into agenda-scoped summaries and machine-validated records.

6. **Keep tool and protocol contracts explicit.**  
   Ambiguous tool descriptions or ambiguous message formats cause misrouting and drift.

7. **Separate reasoning from execution.**  
   Meetings should produce recommendations and action proposals, not falsely claim work has already been done.

These principles are reflected throughout the implementation below.

## 5. Source Normalization Notes

This document intentionally resolves and normalizes differences between sources:

- The uploaded meeting documents strongly prefer **round-robin + interrupt**, hard output tags, open-question integration, agenda-state authority, and action gating.
- The existing `agent-protocol.md` and `workflow-developer-api.md` define structured outcomes and a separate workflow/action system. This spec does **not** replace those designs; it integrates with them safely.
- External guidance often describes several orchestration patterns: subagents, handoffs, supervisors, group chat, maker-checker, human approvals. This spec selects a **strict moderated meeting protocol** because that is the best fit for your feature and your failure modes.

The selected meeting architecture is therefore:

- **Moderator-led, agenda-scoped, round-robin discussion**
- **Limited inline interrupts only**
- **Human decisions recorded explicitly**
- **Strictly typed outputs**
- **Structured summary after each agenda item**
- **Optional post-decision action-item emission to the existing action system**
- **No autonomous workflow delegation from within meetings unless explicitly invoked by another feature**

## 7. High-Level Feature Definition

An **Agent Meeting** is a structured, agenda-driven deliberation session involving:
- one human decider,
- one moderator agent,
- zero or more specialist agents,
- one or more agenda items,
- persistent meeting state,
- structured round outputs,
- moderator summaries,
- optional open question creation,
- optional action proposal creation.

The meeting engine governs:
1. who may speak,
2. what message types are allowed,
3. what state transitions are legal,
4. when human input is required,
5. when an agenda item is closed, deferred, blocked, or ready for human decision,
6. and how meeting artifacts are created.

## 8. Core Design Principles

### 8.1 Human decides
Agents may never finalize decisions. They may only:
- present options,
- recommend,
- identify risks,
- ask clarifying questions,
- validate feasibility,
- propose actions.

A decision becomes effective only when an explicit human decision event is recorded.

### 8.2 Agenda-scoped context
Each agenda item has isolated context.  
Agents discussing item B must not receive the free-form round history of item A except for a very short meeting-level state header when required.

### 8.3 Typed communication
Every agent utterance must be classifiable into one of a small number of allowed output tags.

### 8.4 No fake progress
Meetings produce **plans**, **recommendations**, **questions**, and **action proposals**.  
Agents must not claim artifacts were created, approved, or updated unless the system actually performed that action.

### 8.5 Structured summaries over transcript dependence
The durable state of a meeting is carried by:
- validated rounds,
- structured moderator summaries,
- open-question records,
- decision records,
- action proposal records,
not by replaying raw transcript text.

### 8.6 Bounded interrupt model
Interrupts exist only to resolve blocking questions or request targeted validation.  
Interrupts do not become parallel uncontrolled discussion.

### 8.7 Minimal necessary orchestration
A meeting is not a workflow.  
The orchestration pattern is a strict moderator + round-robin model because this is easier to control and fits your existing product boundaries.

## 9. User Stories

### 9.1 Human decider
- As a user, I want several specialist agents to discuss a design problem without drifting into generic chatter.
- As a user, I want a clear summary after each agenda item with options, risks, open questions, and a direct decision prompt.
- As a user, I want to explicitly decide, defer, or block.
- As a user, I want action items to be generated only when appropriate and only in a way that aligns with my decisions.

### 9.2 Moderator agent
- As a moderator, I need structured meeting state so I can summarize the current agenda item and stop loops.
- As a moderator, I need to know when the agenda item is blocked, deferred, ready for human decision, or resolved.

### 9.3 Specialist agent
- As a specialist agent, I need clear role instructions, per-item context, and strict allowed output forms.
- As a specialist agent, I need a reliable way to request brief targeted validation or ask a blocking question.

### 9.4 Developer
- As an implementer, I need deterministic schemas, event flows, validation gates, and recoverable local state.

## 10. Functional Requirements

## 10.1 Meeting lifecycle

### FR-ML-001 Create meeting
The system shall allow the user to create a meeting with:
- title,
- purpose,
- human decider identity,
- participants,
- moderator,
- agenda items,
- meeting policies,
- file paths for minutes/open questions if configured.

### FR-ML-002 Start meeting
The system shall initialize meeting state, set the first agenda item to active, and emit a meeting-start event.

### FR-ML-003 Pause and resume
The system shall support pausing and resuming meetings without losing structured state.

### FR-ML-004 Close meeting
The system shall allow a meeting to be closed manually or automatically after all agenda items reach terminal status.

### FR-ML-005 Recover after restart
The system shall restore active meetings on extension restart and indicate whether manual attention is required.

---

## 10.2 Agenda control

### FR-AG-001 Explicit active agenda item
The system shall maintain exactly one `activeAgendaItemId` at a time.

### FR-AG-002 Agenda item status model
Each agenda item shall have one of the following statuses:
- `pending`
- `active`
- `needs_human_decision`
- `blocked_by_human`
- `deferred`
- `resolved`
- `cancelled`

### FR-AG-003 Agenda switch authority
The orchestrator shall only run agent turns against the currently active agenda item.

### FR-AG-004 Off-topic suppression
If an agent response materially addresses a non-active agenda item, the orchestrator shall reject and reprompt it or mark it as `[SKIP]`.

### FR-AG-005 Item close record
When an agenda item closes or is deferred, the system shall write a machine-readable close record.

### FR-AG-006 User-driven next item
When the user requests “next agenda item,” the system shall:
1. finalize the current item status if needed,
2. switch `activeAgendaItemId`,
3. rebuild prompts for the new item,
4. and ensure prior free-form history from old items is excluded from the new item context.

---

## 10.3 Meeting roles

### FR-RL-001 Moderator role
Each meeting shall designate exactly one moderator agent.

### FR-RL-002 Human decider
Each meeting shall designate one human decider label.  
All decision prompts shall target this identity.

### FR-RL-003 Specialist roles
Each agent shall expose:
- `agentId`
- display name
- role
- provider/model config
- knowledge configuration
- optional permissions and tools

### FR-RL-004 Optional observer mode
The system may support non-speaking observer agents in future versions, but this is not required for v1.

---

## 10.4 Typed communication protocol

### FR-CP-001 Allowed output tags
Every non-system agent message in a meeting shall begin with exactly one allowed tag:

- `RECOMMENDATION:`
- `RISK:`
- `OPEN_QUESTION:`
- `ACTION:`
- `VALIDATION:`
- `CLARIFICATION_FOR_HUMAN:`
- `[SKIP]:`

### FR-CP-002 Invalid tag handling
If a response does not match the protocol, the system shall reprompt the agent to rewrite its response into a valid format.

### FR-CP-003 Structured payload requirements
Each tag has required content fields:

#### `RECOMMENDATION:`
Must include:
- options,
- recommended option,
- tradeoffs,
- what the human must decide.

#### `RISK:`
Must include:
- risk,
- impact,
- mitigation or follow-up.

#### `OPEN_QUESTION:`
Must include:
- question,
- why it matters / what it blocks,
- example acceptable answer.

#### `ACTION:`
Must include:
- owner role,
- task description,
- acceptance criteria / definition of done.

#### `VALIDATION:`
Must include:
- claim under validation,
- validation result,
- basis or verification method.

#### `CLARIFICATION_FOR_HUMAN:`
Must include:
- precise question,
- why it is blocking,
- acceptable answer shape.

#### `[SKIP]:`
Must include:
- one-line reason for no impactful contribution.

### FR-CP-004 Talkshop prohibition
The protocol shall explicitly forbid:
- greetings,
- thanking,
- vague “building on previous point” phrasing,
- generic best practices not tied to a specific agenda output,
- restating previous speakers unless correcting a factual error.

---

## 10.5 Round-robin protocol

### FR-RR-001 Strict turn order
The system shall run participant turns in a deterministic order for each discussion pass.

### FR-RR-002 Moderator turn placement
The moderator summary shall occur after each full participant round or after explicit item-close triggers.

### FR-RR-003 Skip allowed
Agents may skip without penalty if they have no impactful contribution.

### FR-RR-004 Max rounds per item
The system shall enforce configurable maximum rounds per agenda item.

### FR-RR-005 Loop breaker
When repeated content exceeds a configurable threshold, the moderator shall be triggered to summarize, convert unknowns to open questions, and produce a decision prompt or defer/block state.

---

## 10.6 Interrupt protocol

### FR-IN-001 Allowed interrupt types
Only two interrupt categories are allowed:
1. blocking question to a specific participant,
2. targeted validation request.

### FR-IN-002 Interrupt request syntax
Interrupts shall only be recognized when formatted as:

```text
INTERRUPT_REQUEST: @AgentName — <one precise question> — Context: <1–2 lines>
```

### FR-IN-003 Interrupt limits
Each turn may contain at most two interrupt requests.

### FR-IN-004 Blocking requirement
Interrupts must be genuinely blocking for the caller’s current output.  
If not blocking, the agent must convert the issue into `OPEN_QUESTION:` instead.

### FR-IN-005 Interrupt response constraint
The interrupted agent shall respond only to the asked question and may not introduce new agenda topics.

### FR-IN-006 No turn consumption
Interrupt responses shall not consume the interrupted agent’s normal round-robin turn.

### FR-IN-007 Optional addendum
The original caller may add a brief `ADDENDUM:` after the interrupt response.

---

## 10.7 Human decision and approval model

### FR-HD-001 No implicit decisions
The system shall not infer decisions from agent recommendations.

### FR-HD-002 Durable decision record
A decision becomes effective only when a human decision event is recorded, for example:

```ts
interface HumanDecision {
  decisionId: string;
  meetingId: string;
  agendaItemId: string;
  chosenOption: string;
  rationale?: string;
  decidedBy: string;
  decidedAt: string;
}
```

### FR-HD-003 Decision gating
If no decision record exists for an agenda item, agents may not:
- mark options as accepted,
- mark ADRs or artifacts as approved,
- close related open questions as resolved,
- or update requirements to reflect final adoption.

### FR-HD-004 Decision prompt
Every moderator summary for an unresolved item shall end with an explicit decision prompt for the human.

### FR-HD-005 Defer support
The human may defer a decision.  
If deferred, the system shall record the prerequisites needed before returning to the item.

---

## 10.8 Open Questions integration

### FR-OQ-001 Automatic OQ creation
Whenever an agent emits `OPEN_QUESTION:`, the system shall create an open-question record.

### FR-OQ-002 Durable file append
If meeting configuration includes an open questions file, the system shall append the OQ using a stable template.

### FR-OQ-003 OQ schema
```ts
interface OpenQuestionRecord {
  id: string; // OQ-00001
  meetingId: string;
  agendaItemId: string;
  askedByAgentId: string;
  question: string;
  whyItMatters: string;
  exampleAnswer: string;
  answer?: string;
  status: "open" | "answered" | "cancelled";
  createdAt: string;
  answeredAt?: string;
}
```

### FR-OQ-004 OQ reference in summaries
Moderator summaries shall list OQ IDs created or referenced for the current agenda item.

### FR-OQ-005 Human answer ingestion
If the human answers an open question inline, the system shall update the OQ record and make the answer available to later turns.

---

## 10.9 Action proposals and action-item interoperability

### FR-AI-001 Meetings produce action proposals, not workflow execution
Meetings may generate action proposals, but they shall not directly execute workflow tasks unless another feature explicitly invokes that integration.

### FR-AI-002 Action gating policy
The system shall support meeting-level and item-level action policies:

```ts
type ActionPolicy =
  | { mode: "NORMAL" }
  | { mode: "BLOCK_ALL_ACTIONS" }
  | { mode: "ALLOW_ONLY"; allowed: AllowedActionRule[] };

interface AllowedActionRule {
  tag: "ACTION";
  ownerRole?: string;
  ownerAgentId?: string;
  artifactPath?: string;
  taskPattern?: string;
}
```

### FR-AI-003 Policy enforcement
If an agent emits an `ACTION:` that violates the current policy, the orchestrator shall reject and reprompt or rewrite to `[SKIP]:`.

### FR-AI-004 Human-controlled emission
Optionally, after a human decision, the system may convert selected meeting action proposals into action items in the separate action subsystem.

### FR-AI-005 Action deduplication
Duplicate actions shall be detected and either merged or suppressed.

### FR-AI-006 Action proposal registry
The meeting system shall maintain an internal registry of action proposals to avoid repeated creation.

---

## 10.10 Moderator summaries

### FR-MS-001 Structured moderator summary
The moderator shall produce a schema-valid summary after each full round or item close.

### FR-MS-002 Required fields
```ts
type AgendaResolutionStatus =
  | "needs_human_decision"
  | "blocked_by_human"
  | "deferred"
  | "resolved";

interface ModeratorSummary {
  summaryId: string;
  meetingId: string;
  agendaItemId: string;
  problem: string;
  options: string[];
  recommendation?: string;
  risks: string[];
  actions: ActionProposalRef[];
  openQuestionIds: string[];
  decisionPrompt: string;
  status: AgendaResolutionStatus;
  timestamp: string;
}
```

### FR-MS-003 Parseable format
The summary shall be generated in strict JSON or another fully parseable structured format.

### FR-MS-004 Validation and retry
If the summary is malformed, duplicated, or missing required fields, the orchestrator shall reprompt the moderator.

### FR-MS-005 Loop closure
If discussion is looping, the moderator shall stop additional free discussion and force one of:
- `needs_human_decision`
- `blocked_by_human`
- `deferred`

---

## 10.11 Context assembly and knowledge use

### FR-CTX-001 Per-agent knowledge retrieval
Before each turn, the system shall assemble:
- agenda-item context,
- relevant previous summary for the same item,
- current known assumptions,
- unresolved open questions,
- explicit human decisions,
- selected per-agent knowledge retrieved from the agent’s folders/vector store.

### FR-CTX-002 No global transcript dump
The system shall not feed the full historical meeting transcript to every turn by default.

### FR-CTX-003 Agenda-scoped history
By default, only:
- current item rounds,
- current item summaries,
- and small meeting-level state
shall be included.

### FR-CTX-004 Context budget
The system shall enforce per-turn token budgets and degrade gracefully by:
1. keeping structured state,
2. dropping verbose prior rounds,
3. retaining only the latest relevant summary and unresolved facts.

### FR-CTX-005 Retrieval strategy
Knowledge retrieval shall combine:
- lexical search,
- vector search,
- role-based folder filtering,
- and optional file/path constraints.

### FR-CTX-006 Evidence-first prompts
Where feasible, context passed to agents shall prioritize concise extracted evidence over large raw documents.

---

## 10.12 UI requirements

### FR-UI-001 Single primary meeting view
The meeting UI shall present the current agenda item, participants, current round state, and structured outputs clearly.

### FR-UI-002 Transcript duplication avoidance
If the transcript is already present in chat, the meeting UI shall avoid redundant duplication.

### FR-UI-003 Mention handling
When a human or agent requests a targeted mention, the UI and orchestrator shall reliably route that request and visibly show the pending response state.

### FR-UI-004 Clear agenda state
The UI shall display:
- active agenda item,
- item status,
- blocked/deferred/decision-needed state,
- open questions count,
- action policy state,
- and pending human decisions.

### FR-UI-005 Action buttons
The UI shall not show duplicate action buttons for the same operation.

### FR-UI-006 Resolve behavior
The “resolve” action shall perform a real state transition and not be a no-op.

### FR-UI-007 Summary-first presentation
The UI shall privilege structured summaries over raw verbose transcript.

### FR-UI-008 Inline interrupt rendering
Interrupt responses shall appear inline beneath the requesting agent’s turn.

---

## 10.13 Storage and persistence

### FR-ST-001 Structured storage
The system shall persist meetings as structured state objects, not only as rendered markdown.

### FR-ST-002 Optional markdown exports
The system may maintain human-readable markdown minutes and open question files.

### FR-ST-003 Event sourcing
The system should persist append-only meeting events to simplify replay and debugging.

### FR-ST-004 Consistent IDs
Meetings, rounds, summaries, open questions, decisions, and action proposals shall all use stable IDs.

---

## 10.14 Telemetry and observability

### FR-OBS-001 Per-turn telemetry
The system shall record:
- agent,
- tag,
- agenda item,
- token counts if available,
- retries,
- interrupts,
- validation failures,
- latency,
- and final accepted content length.

### FR-OBS-002 Productivity metrics
The system shall calculate:
- skip ratio,
- duplicate-action suppression count,
- off-topic rejection count,
- average rounds to summary,
- average time to human decision,
- open questions created/answered,
- and agenda completion rate.

### FR-OBS-003 Auditability
All human decisions, state changes, and action-policy changes shall be auditable.

---

## 10.15 Security and safety

### FR-SEC-001 Prompt injection scanning
Meeting content ingested from agents or retrieved knowledge should be scanned similarly to existing injection detection practices.

### FR-SEC-002 Tool boundary respect
Agents in meetings shall not gain broader tool permissions than their configured runtime permissions.

### FR-SEC-003 Decision spoofing prevention
A human decision event must originate from a trusted user action path, not an agent-generated message.

### FR-SEC-004 File write boundaries
Markdown append operations for minutes/open questions shall be limited to configured safe paths.

---

## 11. Non-Functional Requirements

### NFR-001 Determinism
Given identical state, policy, and model outputs, the orchestrator behavior should be deterministic.

### NFR-002 Recoverability
The system must recover active meeting state after restart with no silent loss.

### NFR-003 Responsiveness
UI updates should stream or incrementally render per turn.

### NFR-004 Local-first reliability
The feature must function without a mandatory remote coordinator.

### NFR-005 Extensibility
The protocol must allow future addition of observer roles, voting modes, and richer structured outputs without breaking v1.

### NFR-006 Token economy
The design must minimize unnecessary context fan-out.

### NFR-007 Debuggability
A developer must be able to inspect why a round was rejected, rewritten, skipped, blocked, or summarized.

### MeetingController
- receives UI intents,
- creates/starts/pauses/resumes meetings,
- advances agenda,
- records human decisions,
- applies action policy changes.

### MeetingOrchestrator
- executes rounds,
- enforces turn order,
- handles interrupts,
- coordinates validation/retries,
- triggers moderator summaries.

- constructs per-turn prompts,
- injects protocol rules,
- injects agenda-item-only state,
- keeps token budgets.

- enforces allowed tags and action gating,
- enforces human decision constraints,
- enforces off-topic rejection.

- builds minimal effective context from:
  - agenda item state,
  - summaries,
  - decisions,
  - OQs,
  - per-agent retrieval.

- fetches evidence from per-agent folders/vector store.

- parses, persists, updates, and appends OQs.

### ActionProposalService
- stores proposals,
- deduplicates,
- emits optional action items to the separate subsystem when authorized.

### ModeratorSummaryService
- requests, validates, and stores structured summaries.

### MeetingStorage
- persists state/events,
- supports recovery,
- supports markdown append.

## 17. Parsing and Validation

## 17.1 Tag parser

```ts
const ALLOWED_TAGS = new Set([
  "RECOMMENDATION",
  "RISK",
  "OPEN_QUESTION",
  "ACTION",
  "VALIDATION",
  "CLARIFICATION_FOR_HUMAN",
  "SKIP",
]);

export function parseOutputTag(response: string): OutputTag | null {
  const trimmed = response.trimStart();

  if (/^\[SKIP\]\s*:?/i.test(trimmed)) {
    return "SKIP";
  }

  const match = trimmed.match(/^([A-Z_]+):/);
  if (!match) return null;

  const candidate = match[1].toUpperCase();
  return ALLOWED_TAGS.has(candidate) ? (candidate as OutputTag) : null;
}
```

## 17.2 Interrupt request extractor

```ts
export function extractInterruptRequests(
  response: string,
  participants: Array<{ agentId: string; displayName: string }>
): InterruptRequest[] {
  const requests: InterruptRequest[] = [];
  const lines = response.split(/\r?\n/);

  for (const line of lines) {
    if (!line.includes("INTERRUPT_REQUEST:")) continue;

    const match = line.match(
      /INTERRUPT_REQUEST:\s*@([^—-]+)\s*[—-]\s*(.+?)(?:\s*[—-]\s*Context:\s*(.+))?$/i
    );
    if (!match) continue;

    const display = match[1].trim().toLowerCase();
    const found = participants.find(
      p => p.displayName.toLowerCase() === display || p.agentId.toLowerCase() === display
    );
    if (!found) continue;

    requests.push({
      mentionedAgentId: found.agentId,
      question: match[2].trim(),
      context: match[3]?.trim(),
    });

    if (requests.length >= 2) break;
  }

  return requests;
}
```

## 17.3 Structured payload parser examples

```ts
export interface ParsedRecommendation {
  options: string[];
  recommendedOption: string;
  tradeoffs: string[];
  humanDecisionRequired: string;
}

export function parseRecommendationBody(body: string): ParsedRecommendation | null {
  const options = [...body.matchAll(/^\s*[-*]\s*Option\s+(.+)$/gim)].map(m => m[1].trim());
  const recommendedMatch = body.match(/Recommended option\s*:\s*(.+)$/im);
  const tradeoffs = [...body.matchAll(/^\s*[-*]\s*Tradeoff\s*:\s*(.+)$/gim)].map(m => m[1].trim());
  const decisionMatch = body.match(/What the human must decide\s*:\s*(.+)$/im);

  if (!options.length || !recommendedMatch || !decisionMatch) return null;

  return {
    options,
    recommendedOption: recommendedMatch[1].trim(),
    tradeoffs,
    humanDecisionRequired: decisionMatch[1].trim(),
  };
}
```

## 18.2 Human decision constraint check

```ts
export function assertNoImplicitDecision(
  text: string,
  hasHumanDecision: boolean
): string | null {
  if (hasHumanDecision) return null;

  const forbidden = [
    /\baccepted\b/i,
    /\bapproved\b/i,
    /\bfinal decision\b/i,
    /\bmark(ed)? .* as accepted\b/i,
    /\bupdate requirements to reflect the approved\b/i,
  ];

  return forbidden.some(rx => rx.test(text))
    ? "Implicit or unauthorized final decision detected without recorded human decision."
    : null;
}
```

## 18.3 Action policy check

```ts
export function isActionAllowed(
  policy: ActionPolicy | undefined,
  proposal: { ownerRole?: string; ownerAgentId?: string; task?: string }
): boolean {
  if (!policy || policy.mode === "NORMAL") return true;
  if (policy.mode === "BLOCK_ALL_ACTIONS") return false;

  return policy.allowed.some(rule => {
    if (rule.ownerRole && rule.ownerRole !== proposal.ownerRole) return false;
    if (rule.ownerAgentId && rule.ownerAgentId !== proposal.ownerAgentId) return false;
    if (rule.taskPattern && proposal.task && !new RegExp(rule.taskPattern, "i").test(proposal.task)) {
      return false;
    }
    return true;
  });
}
```

## 19. Orchestrator Flow

## 19.1 Main run loop

```ts
export async function runMeeting(
  meetingId: string,
  deps: OrchestratorDeps
): Promise<void> {
  const meeting = await deps.storage.loadMeeting(meetingId);
  if (!meeting) throw new Error(`Meeting not found: ${meetingId}`);

  for (const item of meeting.agenda) {
    if (item.status === "resolved" || item.status === "cancelled") continue;

    await setActiveAgendaItem(meeting, item.id, deps);
    await runAgendaItem(meeting, item.id, deps);

    const updated = await deps.storage.loadMeeting(meetingId);
    if (!updated) throw new Error(`Meeting disappeared during run: ${meetingId}`);
    if (updated.status !== "active") break;
  }
}
```

## 19.2 Activate agenda item

## 19.3 Run one agenda item

```ts
async function runAgendaItem(
  meeting: MeetingState,
  agendaItemId: AgendaItemId,
  deps: OrchestratorDeps
): Promise<void> {
  const item = meeting.agenda.find(a => a.id === agendaItemId)!;

  for (let roundIndex = 1; roundIndex <= meeting.config.maxRoundsPerAgendaItem; roundIndex++) {
    item.currentRoundIndex = roundIndex;
    await deps.storage.saveMeeting(meeting);

    for (const agentId of meeting.config.participantAgentIds) {
      await runAgentTurn(meeting, item, agentId, deps);

      const refreshed = await deps.storage.loadMeeting(meeting.id);
      if (!refreshed) throw new Error("Meeting state lost");
      const refreshedItem = refreshed.agenda.find(a => a.id === agendaItemId)!;

      if (["needs_human_decision", "blocked_by_human", "deferred", "resolved"].includes(refreshedItem.status)) {
        break;
      }
    }

    await generateModeratorSummary(meeting.id, agendaItemId, deps);

    const latest = await deps.storage.loadMeeting(meeting.id);
    const current = latest?.agenda.find(a => a.id === agendaItemId);
    if (!current) throw new Error("Agenda item missing after summary");

    if (["needs_human_decision", "blocked_by_human", "deferred", "resolved"].includes(current.status)) {
      return;
    }
  }

  // Hard stop on max rounds
  await forceLoopBreakerSummary(meeting.id, agendaItemId, deps);
}
```

## 19.4 Run agent turn

```ts
async function runAgentTurn(
  meeting: MeetingState,
  item: AgendaItem,
  agentId: AgentId,
  deps: OrchestratorDeps
): Promise<void> {
  const context = await deps.contextAssembler.build({
    meetingId: meeting.id,
    agendaItemId: item.id,
    agentId,
  });

  const prompt = deps.promptBuilder.buildTurnPrompt({ meeting, item, agentId, context });

  let attempt = 0;
  while (attempt < 3) {
    attempt++;

    const response = await deps.provider.runAgent({
      meetingId: meeting.id,
      agentId,
      prompt,
    });

    const accepted = await deps.policyEngine.validateTurn({
      meeting,
      item,
      agentId,
      responseText: response.text,
    });

    if (!accepted.ok) {
      prompt.push({
        role: "system",
        content: `Rewrite required. Reason: ${accepted.reason}`,
      });
      continue;
    }

    const round = await deps.roundFactory.createAcceptedRound({
      meeting,
      item,
      agentId,
      responseText: accepted.normalizedText,
      retryCount: attempt - 1,
    });

    await deps.storage.appendEvent({
      type: "ROUND_ACCEPTED",
      roundId: round.id,
      meetingId: meeting.id,
      agendaItemId: item.id,
      tag: round.tag,
      at: new Date().toISOString(),
    });

    await handleAcceptedRound(round, deps);
    return;
  }

  await deps.storage.appendEvent({
    type: "ROUND_REJECTED",
    meetingId: meeting.id,
    agendaItemId: item.id,
    agentId,
    reason: "Exceeded rewrite attempts",
    at: new Date().toISOString(),
  });
}
```

## 25. Failure Modes and Required Controls

### FM-001 Implicit decision adoption
**Symptom:** Agents act as if a recommendation is already approved.  
**Control:** hard decision gate tied to durable human decision records.

### FM-002 Agenda drift
**Symptom:** agents continue discussing old agenda items.  
**Control:** strict active-agenda state and agenda-scoped context only.

### FM-003 Action leakage
**Symptom:** agents create actions when actions are blocked.  
**Control:** policy engine with `BLOCK_ALL_ACTIONS` / `ALLOW_ONLY`.

### FM-004 Malformed summaries
**Symptom:** moderator summaries contain broken or duplicated content.  
**Control:** strict JSON schema validation and retry.

### FM-005 Fake implementation claims
**Symptom:** agents say files/ADRs/docs were already created during discussion.  
**Control:** planning-mode wording enforcement.

### FM-006 Mention/interrupt failure
**Symptom:** targeted requests are ignored and conversation continues.  
**Control:** explicit interrupt lifecycle, pending-response tracking, visible UI state.

### FM-007 Open questions not persisted
**Symptom:** questions appear in chat but not in open-question storage.  
**Control:** OQ creation must be atomic with accepted `OPEN_QUESTION:` rounds.

### FM-008 Duplicate actions
**Symptom:** same action repeatedly proposed.  
**Control:** action dedupe registry.

### FM-009 Context poisoning / prompt injection
**Symptom:** retrieved content manipulates meeting flow.  
**Control:** input scanning and strict protocol/validation.

## 30. Implementation Notes for VS Code Extension Context

1. Keep long-running meeting state outside volatile webview state.
2. The source of truth must be extension-host storage, not only frontend memory.
3. Webview messages should be event-based and idempotent where possible.
4. The webview should receive structured events, not only rendered markdown.

Example bridge contracts:

```ts
type WebviewMessage =
  | { type: "meeting/state"; state: MeetingStateViewModel }
  | { type: "meeting/roundAccepted"; round: MeetingRoundViewModel }
  | { type: "meeting/summaryAccepted"; summary: ModeratorSummary }
  | { type: "meeting/openQuestionCreated"; oq: OpenQuestionRecord }
  | { type: "meeting/error"; message: string };
```

## 32. Implementation Recommendations Summary

The implementation team should prioritize these changes first:

1. **Make moderator summaries strict JSON and validate them.**
2. **Enforce agenda-scoped context only.**
3. **Record human decisions explicitly and gate downstream claims on them.**
4. **Implement action-policy enforcement.**
5. **Persist and reference open questions reliably.**
6. **Treat meetings as planning/deliberation, not execution.**
7. **Keep the meeting engine separate from the workflow engine.**

These seven controls address nearly all failure modes documented in the uploaded meeting analyses.

# Implementation Plan
**Feature:** AI Coding Assistant Sandbox Separated Work Environment

## Overview
Because the sandbox environment feature currently has 0% implementation, this plan proposes a sequenced rollout. Attempting to build Containers, Remote Runners, and Worktrees all at once is highly risky. 

In alignment with the spec's rollout recommendation (Section 30), this plan focuses on **Release 1: Local Worktree Sandbox & Policy Enforcement**.

## Phase 1: Core Foundation & Policy Engine (Release 1)
**Goal:** Establish the rules and the execution wrappers before actually branching the filesystem.

1. **Policy Engine (`src/sandbox/PolicyEngine.ts`)**
   - Implement single-layer project policy resolution (reading `.bormagi/policies/sandbox.policy.yaml`).
   - Author default YAML policies for `standard-coding` and `secure-review`.
   - Implement Path matching rules (Allowlist/Denylist).
   - Implement Command classification logic (`HIGH_RISK_PATTERNS`, `allow`, `ask`, `deny`).

2. **Upgraded Approval Model (`src/sandbox/ApprovalService.ts`)**
   - Refactor `ui/ApprovalDialog.ts` to support the new `ApprovalScope` (`once`, `task`, `session`, `project`).
   - Store session and project-scoped approvals in `.bormagi/approvals.json`.

3. **Execution & Tool Mediation Wrapper (`src/sandbox/ExecWrapper.ts`)**
   - Ensure the MCP `run_command` tool is routed through `guardedCommand()`.
   - Add STDOUT/STDERR secret redaction logic.

## Phase 2: Workspace Isolation (Release 1 continued)
**Goal:** Move the agent into an isolated folder.

1. **Sandbox Manager (`src/sandbox/SandboxManager.ts`)**
   - Implement `local_worktree_sandbox` mode using Git Worktrees, with a graceful fallback to a simple directory copy if no Git repository is found.
   - Generate `manifest.json`.
   - Track active tasks in `~/.bormagi/sandboxes/`.
   
2. **Context and Path Redirection**
   - Modify `MCPHost.ts` or individual tools (`read_file`, `write_file`) so that when the agent requests path edits, they are dynamically mapped into the sandbox working directory, ensuring the host files remain untouched.

3. **Git Checkpointing (`src/sandbox/CheckpointService.ts`)**
   - Automatically execute `git commit -m "[checkpoint]"` prior to bulk tool executions.

## Phase 3: Promotion and Review UI
**Goal:** Allow the user to merge the agent's work back to the host smoothly.

1. **Review UX**
   - Leverage standard VS Code diff editors to show changes.
   - Use a Yes/No prompt for applying changes, ensuring an 'Accept All' option is included where relevant for bulk changes.
2. **Promotion Actions**
   - Implement `Apply Patch` and `Merge Worktree` back into the host environment upon user approval.
   - Implement sandbox cleanup workflows.

## Refactoring Impacts
- To keep the implementation simple and centralized, interception of tools (`run_command`, `write_file`) will be done at the `MCPHost` or `AgentRunner` level before reaching the actual MCP servers. This avoids having to duplicate `PolicyEngine` logic across multiple independent servers.
- `AgentRunner.ts` will need initialization updates to request a Sandbox environment at the start of a task.

## Automated Testing Strategy
- **Unit Tests:** `PolicyEngine` matrix evaluations, `ApprovalService` cache lifecycle, `ExecWrapper` regex and redaction tests.
- **Integration Tests:** End-to-end sandbox creation, editing a file safely in the sandbox, and verifying the host file is untouched.

## Rollout Order
1. Implement and test Policy Engine + Approvals (Unit tests).
2. Implement ExecWrapper + Secret Redaction.
3. Hook existing MCP tools into the Policy Engine without isolation (Dry run safety).
4. Implement SandboxManager and `local_worktree_sandbox`.
5. Implement Promotion UX and Workspace mapping.

# Implementation Status Report
## Feature: AI Coding Assistant Chatbot AI / UX
### Spec: `7.ai_coding_assistant_chatbot_ai_ux_spec.md`
### Date: 2026-03-08

---

## Executive Summary

The codebase contains a **well-structured VS Code extension** (`bormagi-extension`) with a strong backend foundation ‚Äî including mode classification, plan management, context compaction, checkpoint management, instruction resolution, and sandbox policy enforcement. However, **the majority of the chat UX requirements specified in ¬ß5‚Äì¬ß26 have not been surfaced to the user** in the front-end webview (`media/chat.html`) or wired into the `ChatController ‚Üí webview` contract.

The implementation is **Phase 1 incomplete** per the spec's own rollout plan (¬ß28). Core foundational UX elements ‚Äî mode switcher in the composer, structured approval prompts, message class differentiation, action timeline with tool cards, and a context rail ‚Äî are absent from the chat UI. The backend services that would power these features exist and are well-implemented; the gap is almost entirely front-end integration and wiring.

**Estimated overall implementation completeness: ~30%** (backend ~65%, front-end ~15%)

## Unwired / Incomplete Planned Items Found

1. **Mode picker referenced but absent from `chat.html`**: `ModeClassifier.ts` comment at line 6 says "The user selects mode explicitly via the status bar dropdown (OQ-1 answer: C)" ‚Äî but `chat.html` has no mode dropdown. The `StatusBar.ts` exists but its mode dropdown is not wired into the composer or chat header.

2. **`ContextCompactor` fires but sends no notification**: `ContextCompactor.ts` compacts history; `CompactionOutput` is returned but `ChatController` does not post a `compaction_notice` message to the webview.

3. **`InstructionResolver` resolves layers but result never shown to user**: `EffectiveInstructions` with `layers[]` is built but never surfaced as a context panel item.

4. **`PlanManager` creates plans to disk but no plan card in chat**: `createPlan()` is called from `AgentRunner` but the created `ExecutionPlan` is never transmitted to the webview for rendering.

5. **`PolicyEngine` has approval levels but no UI toggle**: `PolicyEngine.ts` enforces policy but the user cannot see or change the current autonomy profile from the chat.

6. **`CheckpointManager` emits checkpoints but only `checkpointId` in `git_status`**: The checkpoint pill shows the ID but there is no inline checkpoint creation notice in the message stream.

7. **`CapabilityRegistry` / `SkillManager` loads skills but no `/skill <name>` command**: Skills are loaded on startup but not discoverable or invocable from the chat composer.

8. **`action_request` card lacks structured fields**: The spec requires `ApprovalRequest { kind, reason, scope, risk, alternatives }` but the current `action_request` message only carries `{ id, prompt: string, actions: string[] }`.

9. **Empty state has no concrete examples** per ¬ß16.2: current text is generic ("No conversation yet").

10. **No `AssistantTurnEnvelope`**: Every response is streamed text. The backend never emits structured `artifacts`, `actions`, `context`, or `warnings` in a turn envelope.
